# Validate Chain — 验证新实现与旧实现的行为等价性

## 目标

对比旧版 C++ 实现和新版 C 实现，验证**保留功能（inventory 中标记 "keep" 的功能）**
在相同输入下产出相同行为。

**"等价"的定义：** 相同输入 → 相同语义输出。不要求代码结构相同、变量名相同、
甚至中间表示相同——只要求最终对外可观测的行为一致。

## 前置条件

- 可访问旧版 C++ 代码库
- 可访问新版 C 代码库（`src/` 目录）
- 已完成 Discover 链，拥有结构化功能清单（Inventory JSON）
- Inventory 中每个功能已标注 `keep` / `remove` / `uncertain`

## 核心原则

### 1. 只验证 "keep" 功能

Inventory 中标记 `remove` 的功能是**有意删除的**。如果新代码中没有这些功能，
这是正确行为，不是 bug。

### 2. 实现方式可以不同

旧代码用 C++ class + virtual，新代码用 C struct + 函数指针。
旧代码用异常，新代码用错误码。这些都是**可接受的差异**。

### 3. 差异分为可接受和不可接受

| 类别 | 示例 | 判定 |
|------|------|------|
| 可接受 | 输出空白差异、hex 大小写、字段顺序 | PASS |
| 可接受 | 新代码对边界情况有更好的处理 | PASS |
| 不可接受 | 值错误、字段缺失、类型处理遗漏 | FAIL |
| 不可接受 | keep 功能在新代码中完全缺失 | FAIL |

## 链结构

| 步骤 | 文件 | 做什么 |
|------|------|--------|
| 1 | 01-api-surface-diff.md | 对比公开 API：匹配/删除/遗漏/新增 |
| 2 | 02-type-coverage-check.md | 验证 DWARF 类型处理的覆盖率 |
| 3 | 03-output-comparison.md | 运行对比：相同变量，diff 格式化输出 |
| 4 | 04-edge-case-audit.md | 边界情况审计 |
| 5 | 05-validation-report.md | 汇总 pass/fail 报告 |

## 预期产出

一份结构化的验证报告（Markdown + JSON），包含：
- 每项检查的 PASS / FAIL / SKIP 状态
- FAIL 项的具体证据（输入、预期输出、实际输出）
- SKIP 项的理由（通常是 "intentionally removed"）
- 总体可信度评估

## 常见错误

- **把有意删除当 bug 报**——先查 Inventory 再下结论
- **要求代码结构相同**——我们只比行为，不比实现
- **忽略新增功能**——新代码可能比旧代码多了改进，这是好事，记录但不判 FAIL
