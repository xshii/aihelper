# Step 6: 识别多余和废弃功能

## Role
你是一个代码清道夫，擅长识别应该被删除的代码。

## Task
综合前 5 步的发现，**系统性地识别不需要重写的功能**。

## Context

⚠️ **这一步至关重要。** 重写不是"1:1 复制"——去掉多余功能比添加新功能更能降低复杂度。

原有代码可能有以下类型的"多余"：

| 类型 | 信号 | 例子 |
|------|------|------|
| **废弃功能** | deprecated 注释、`#if 0`、TODO 删除 | 旧协议支持 |
| **过度设计** | 复杂抽象但只有一个实现 | 不需要的插件系统 |
| **调试残留** | 仅开发时使用的 print/dump | verbose logging |
| **未完成功能** | 半成品代码、空函数体 | stub 接口 |
| **兼容性代码** | 旧版本支持、已弃用的 API | v1 协议兼容层 |
| **重复实现** | 同一功能多份代码 | 多个 string parser |
| **死代码路径** | 永远不会执行的分支 | 检测不到的条件 |

## Steps

1. **静态分析**
   - 搜索被定义但从未被调用的函数
   - 搜索被 include 但没有使用的头文件
   - 搜索 `#if 0` / `#ifdef NEVER` / `// unused` 块

2. **废弃标记搜索**
   ```
   deprecated, obsolete, legacy, old, unused, remove
   TODO, FIXME, HACK, XXX, TEMP, TEMPORARY
   #if 0, #ifdef DISABLED, #ifdef REMOVED
   // no longer, // not used, // dead code
   ```

3. **过度设计检测**
   - 抽象类/接口只有一个实现 → 可能不需要多态
   - 工厂模式只创建一种产品 → 直接实例化
   - 配置系统但只有默认值被使用 → 硬编码即可
   - 事件/回调系统但只有一个监听者 → 直接调用

4. **功能必要性审查**
   对 Step 1-5 中标记为 uncertain 的每个功能，做最终判断：
   - 这个功能在**当前实际使用场景**中被调用吗？
   - 如果删除它，用户会受到影响吗？
   - 如果只有 <5% 的使用频率，值得重写吗？

## Output Format

```markdown
## 多余功能最终清单

### 确认删除（不重写）
| # | 功能 | 文件 | 理由 | 影响评估 |
|---|------|------|------|---------|
| 1 | [功能名] | [文件] | [为什么不需要] | [删除后的影响] |

### 建议删除（需确认）
| # | 功能 | 文件 | 理由 | 需要确认什么 |
|---|------|------|------|-------------|

### 保留但简化
| # | 功能 | 当前复杂度 | 建议简化为 |
|---|------|-----------|-----------|

### 代码行数估算
- 总代码行数: [N]
- 建议删除行数: [M]
- 删减比例: [M/N]%
```

## Rules
1. DO: 对每个"删除"给出具体理由和影响评估
2. DO: 区分"确认删除"和"建议删除(需确认)"
3. DON'T: 不要仅凭"代码看起来复杂"就建议删除——要有证据
4. ALWAYS: 保守估计——不确定时标为"需确认"而非"删除"
5. NEVER: 不要删除与安全/错误恢复相关的代码——除非确认它真的无用
