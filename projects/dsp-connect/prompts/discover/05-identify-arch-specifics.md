# Step 5: 盘点架构相关代码

## Role
你是一个嵌入式架构专家，擅长识别与处理器架构相关的代码差异。

## Task
找到现有代码中**所有与目标硬件架构相关的代码**。

## Context

不同 DSP/MCU 架构在以下方面可能不同：
- **寻址方式**: 字节寻址（ARM）vs 字寻址（某些 DSP）
- **字节序**: 大端 vs 小端
- **字长**: 16-bit / 24-bit / 32-bit word
- **地址空间**: 多个独立地址空间（程序空间、数据空间）
- **对齐要求**: 不同架构对齐规则不同

## Steps

1. **找架构相关代码**
   搜索关键词：
   ```
   endian, byte_order, swap, bswap
   word_size, word_bits, word_length
   address_space, addr_shift, addr_mask
   alignment, align, aligned, unaligned
   byte_addressed, word_addressed
   arm, cortex, dsp, ceva, c6000, tensilica
   program_space, data_space, io_space
   ```

2. **识别地址转换逻辑**
   - 逻辑地址（DWARF 中的）→ 物理地址（发给硬件的）的转换
   - 是否有地址偏移/移位
   - 多地址空间的处理

3. **记录架构差异点**
   对每个发现的差异：

   | 差异 | 现有处理方式 | 是否硬编码 | 是否需要参数化 |
   |------|-------------|-----------|---------------|
   | 字节序 | | | |
   | 字长 | | | |
   | 地址转换 | | | |
   | 对齐 | | | |

4. **识别硬编码 vs 可配置**
   - 哪些架构参数是硬编码的 magic number
   - 哪些是通过配置传入的
   - 哪些应该但没有被参数化

## Output Format

```markdown
## 架构相关代码清单

### 目标架构特征
- 寻址方式: [字节/字]
- 字长: [N bits]
- 字节序: [大端/小端]
- 地址空间: [单一/多个]

### 架构适配点
| # | 适配点 | 文件:行 | 当前实现 | 是否硬编码 | 建议 |
|---|--------|---------|---------|-----------|------|

### 多余的架构代码
| # | 功能 | 删除理由 |
|---|------|---------|

### ⚠️ 硬编码的 magic number
| # | 值 | 含义 | 文件:行 | 应改为 |
|---|---|------|---------|--------|
```

## Rules
1. DO: 特别关注地址转换——这是最容易出 bug 的地方
2. DO: 记录所有 magic number 及其含义
3. DON'T: 不要假设只有一种目标架构
4. ALWAYS: 标记硬编码值是否需要参数化
