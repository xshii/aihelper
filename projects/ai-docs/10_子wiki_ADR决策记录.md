# 子 Wiki 10：架构决策记录（ADR）

> Architecture Decision Records —— 关键选型的**历史背景 + 选了什么 + 为什么 + 代价 + 何时可以推翻**。
>
> 目的：让后人（包括弱 AI）看到"为什么是这样"，避免 6 个月后"这个设计一定有坑我要改"式的重复错误。

---

## 约定

每条 ADR 格式：

- **状态**：Accepted / Superseded by ADR-XX / Rejected / Proposed
- **上下文**：为什么需要做这个决策（业务 / 技术背景）
- **决策**：最终选了什么
- **代价**：选它的代价是什么
- **可选方案**：考虑过但没选的，以及没选的理由
- **推翻条件**：什么情况下应该回来重新考虑

## ADR-01：Autotest 主体采用 pytest

**状态**：Accepted

**上下文**：Autotest 需要一个主流程编排框架，承载用例收集、参数化、fixture、报告、并行、插件等能力。备选是自建 runner 或沿用内部老测试框架。

**决策**：主体用 **pytest**。所有用例 = pytest 函数；超时 / 重试用装饰器；启动矩阵用 `@parametrize`；环境准备用 fixture；用例分类用 markers；多板并行用 pytest-xdist。

**代价**：
- 框架学习成本（新人要懂 pytest fixture scope、conftest 层级、markers、plugin）
- pytest 跨三层使用，需要约定好什么放在哪一层（已在 06 · 8.1 定义）

**可选方案**：
- 自建 runner：~~太重，生态空白（没有现成 xdist、html 报告、allure、pytest-timeout）~~
- 沿用老测试框架：~~跨平台扩展能力弱，难融入 LAVA / xdist~~

**推翻条件**：当 pytest 成为瓶颈（比如 pytest 对 Autotest 特有场景的扩展不足，且 plugin 也写不出来），才重新考虑。目前不太可能。

**相关**：00 · 4.4.4，06 · 8

## ADR-02：精度裁决采用 bit 级一致，不引入容忍度

**状态**：Accepted

**上下文**：芯片验证的精度比对有两种思路——**bit 级一致**（ASIC 验证传统）或**容忍度**（AI 业界常见，如 cosine / L1 / QSNR 阈值）。

**决策**：**裁决只看 bit 级一致**（PASS / FAIL 两态）。FAIL 时启用辅助分析（分 block / LCS 偏移 / 位域 / QSNR）**只用于定位，不改判**。

**代价**：
- 不能复用业界基于容忍度的 PyTorch / ONNX Runtime 参考
- 跨平台时要保证所有实现真的 bit-exact（综合、优化器、累加顺序都不能引入差异）
- GOLDEN 必须来自硬件级仿真（见 ADR-03）

**可选方案**：
- 用容忍度（cosine > 0.999 等）：~~AI 业界常见，但会把硬件的 bit 级 bug 吞掉——硬件验证本质是"算对不对"，不是"算得差不多对"~~
- 主要用 bit 级，少数场景允许容忍度：~~口子一开就收不住；语义上自相矛盾~~

**推翻条件**：硬件实现从确定性架构转向概率性（如近似计算、TPU 式的低精度合法发散）；现阶段不适用。

**相关**：02 · 7.1，03 · 2.1

## ADR-03：GOLDEN 来自 GC 硬件计算功能仿真

**状态**：Accepted

**上下文**：确定 ADR-02（bit 级裁决）后，GOLDEN 的来源必须能产出"与硬件 bit 级一致的输出"。PyTorch / ONNX Runtime / CPU 参考都做不到（reduction 顺序、中间精度、舍入模式、量化尺度不一致）。

**决策**：GOLDEN 来自 **GC 硬件计算功能仿真**——硬件计算的功能级参考实现，仿真硅片的算子级行为。GOLDEN 以**硬件目标格式**存储（非 double）。每基线的 `gc_version` 必须与被测 FPGA 版本匹配（GC 可能先修 bug，FPGA 还在发布版本时，**继续用发布版对应的 GC**）。

**代价**：
- GOLDEN 生产依赖 GC 团队；GC 出版本节奏影响回归节奏
- 版本匹配约束多（基线 manifest 必须记 `gc_version`）
- GOLDEN 数据量大 → 归档需要 dbox / onebox / git lfs（归档方案还在演进，见 03 · 2.2.4）

**可选方案**：
- CPU / GPU 参考：~~bit 级对不齐，见 ADR-02 分析~~
- 上一版基线作为 golden：~~只验证"没变"不验证"对"；初始版本没基线~~

**推翻条件**：GC 仿真不可用或无法按需出版本时需要找替代。不太可能。

**相关**：03 · 2.2

## ADR-04：软件架构采用领域分层（放弃洋葱模型）

**状态**：Accepted（Superseded 了最初的洋葱模型方案）

**上下文**：早期 wiki 5 / 6 的架构描述用了**洋葱架构**（核心领域 / 端口 / 服务 / 应用 / 基础设施 5 层）。团队反馈洋葱对当前系统"过于抽象"——本系统的领域核心其实很薄（用例 / 裁决 / 版本几个概念），大部分复杂度在平台适配和编排上。

**决策**：改用**领域分层 3 层**：

```
③ 平台适配（adapter）
    ↓ 依赖端口接口
② 能力服务（精度比对 / dump 策略 / 切换 / 首发散定位 / ...）
    ↓ 依赖领域对象
① 核心领域（用例 / 基线 / 裁决 / 比数模式 / ...）
```

扩展点永远在第 ③ 层，新增平台只写 adapter，上两层零修改。

**代价**：
- 放弃洋葱的某些好处（端口和应用服务的明确分离）；对能力服务层的编排责任有所加重
- 需要约定清楚"哪些逻辑在能力服务、哪些在 adapter"（已在 06 · 5.2 / 5.3 给出允许 / 禁止清单）

**可选方案**：
- 洋葱 5 层：~~领域核心薄，5 层显得拗口；扩展点反而不直观~~
- 平铺（不分层）：~~业务逻辑和平台细节纠缠，新增平台要动全流程~~

**推翻条件**：领域概念变厚（比如引入 DDD 重业务聚合）时可以重新考虑洋葱；目前领域简单。

**相关**：00 · 2.2，06 · 3

## ADR-05：流水线为 link ‖ rtl2c → fpga → emu

**状态**：Accepted

**上下文**：5 个平台的回归编排顺序。最初方案是 "link 准入 → fpga ‖ rtl2c 并行 → emu 性能"。但 rtl2c 执行慢且无法单独承担准入职责（波形采集耗时），同时 rtl2c 失败大概率 fpga 也失败——有**交叉验证价值**。

**决策**：

```
公共前置
   │
   ▼
link ‖ rtl2c （并行：link 准入 + rtl2c 波形定位能力）
   │
link 通过?
   │
  是
   ▼
fpga （主回归）
   │
fpga 通过?
   │
  是
   ▼
emu （性能出口）
```

- **link 准入**：快、卡不合格版本
- **rtl2c 与 link 并行**：提供波形导出能力（FPGA 无此能力）+ 交叉校验
- **fpga 主回归**：link 通过后启动，避免浪费板级资源
- **emu 性能出口**：FPGA 测性能不准

**代价**：
- rtl2c 即使 link 阻断也继续跑（浪费一点 rtl2c 资源，但 rtl2c 资源弹性大）
- fpga 必须等 link 通过，入口点延后几分钟

**可选方案**：
- fpga 和 link 并行：~~link 还没判定就占用板级资源，浪费稀缺资源~~
- fpga 和 rtl2c 并行（去掉 link 准入）：~~准入时间变长到小时级；不合格版本大量浪费板~~
- emu 作为准入：~~emu 比 link 慢；机位稀缺~~

**推翻条件**：link 或 rtl2c 的基础设施发生重大变化（如 link 变慢成为瓶颈）。

**相关**：00 · 4.9，05 · 2.1

## ADR-06：EMU 通过自有 Python API 集成到 pytest

**状态**：Accepted

**上下文**：EMU 有自有 API（走 espi 控制 CPU 桩、TESGINE 灌数据），不走 RDO / 接口 FPGA 链路。接入 Autotest 的工程量是关键不确定性。

**决策**：**EMU 自有 API 提供 Python 接口，直接 `import` 到 pytest fixture 中调用**，不做跨语言 bridge / RPC。adapter 负责把 EMU 原生返回值转成领域对象。

**代价**：
- 工程量降到"写 adapter 包一层"，不用 IPC / 协议设计
- Python 版本和 EMU API 版本耦合（需要 EMU 团队维护稳定的 Python 绑定）

**可选方案**：
- 自建 IPC bridge：~~工程量大，维护成本高~~
- 命令行调用 + 文件交换：~~慢、难处理交互式场景（如 checkpoint）~~

**推翻条件**：EMU 团队停止维护 Python 绑定。

**相关**：00 · 2.2.1，06 · 5.2 / 7.3

## ADR-07：compare 比数模块独立仓库管理

**状态**：Accepted

**上下文**：精准比数 + 辅助分析（分 block / LCS / 位域 / QSNR）既要在 Autotest 内调用（备选比数路径 + 非 FPGA 平台），也要能离线使用（调试分析、回归数据分析）。放在 Autotest 仓里会造成强耦合。

**决策**：**compare 模块独立仓库管理**，与出数工具共仓（因共用底层数据格式、硬件目标类型、GOLDEN 索引规范）。Autotest 以依赖方式引入。

**代价**：
- 多一个仓要维护
- 版本同步——compare 的语义变化要同步到 Autotest 调用

**可选方案**：
- 放在 Autotest 内：~~跨平台 / 离线场景不好用；Autotest 本身很重~~
- 放在业务组件内：~~每个业务组件一份，跨组件分析时不兼容~~

**推翻条件**：compare 只在 Autotest 场景使用（调试离线分析不再需要）。

**相关**：02 · 4.1.2 / 8.4，03 · 2.3.3

## ADR-08：阶段性比数设两条执行路径（标准 / 备选）

**状态**：Accepted

**上下文**：标准路径（被测 FPGA → 接口 FPGA 硬件比对）性能好但依赖消息通道和接口 FPGA 资源；调试期 / 路径未就绪 / 非 FPGA 平台上用不了。

**决策**：阶段性比数**同时支持两条路径**：

- **标准路径**：接口 FPGA 硬件比对（快，零扰动，PASS/FAIL 两态）
- **备选路径**：CPU 桩通过软调底软接口直读被测 FPGA 内存做 dump，由主机侧 compare 模块比对（灵活，产出辅助分析报告，跨平台复用）

**分水岭**：凡 CPU 桩 ↔ 被测 FPGA 的交互都有两条路径，选哪条由"消息通道是否通"决定，Autotest 运行时自动选。

**代价**：
- adapter 要实现两套交互方式
- 用例感知不到路径切换——要靠 adapter 正确实现 fallback 逻辑

**可选方案**：
- 只做标准路径：~~调试场景 / 非 FPGA 平台没法用~~
- 只做备选路径：~~性能差，放弃硬件比对的零扰动优势~~

**推翻条件**：接口 FPGA 硬件比对能力退化或被替代。

**相关**：02 · 4.1

## ADR-09：资源调度优先对标 LAVA 而非自建

**状态**：Proposed（待架构委员会决策）

**上下文**：多 FPGA 板、多 EMU 机位的分配与排队需要**资源调度服务**。有两种选择——引入成熟开源方案（LAVA 等）或自建。

**决策建议**：**优先对标 LAVA**（Linaro Automated Validation Architecture，开源），其 scheduler + device dictionary + multinode job 模型几乎完美对应本场景。只有在 LAVA 引入成本不可接受时才自建最小化调度服务。

**代价**：
- LAVA 学习成本；需要适配本项目的设备定义
- 不完全贴合时要二开

**可选方案**：
- 自建最小网页 + 数据库 + 队列：~~看似简单，长期维护成本高；LAVA 已解决的坑全要自己再踩~~
- TestGrid / Jenkins Lockable Resources：~~调度语义不如 LAVA 贴合~~

**推翻条件**：LAVA 评估后发现引入成本过高（比如和现有 CI 系统冲突严重）。

**相关**：00 · 4.7.5，05 · FR-5 / 开放问题 #9

## ADR-10：LINK 冒烟作为融合出包的必经关卡

**状态**：Accepted

**上下文**：smartci 仓融合两个业务组件 + 资源表后，产出一个完整 DO。融合错了（资源表冲突未检测到 / 出包脚本 bug）会让下游 FPGA / EMU 浪费稀缺资源。

**决策**：smartci 融合出包后**必须在 LINK 平台本地跑一遍冒烟**，通过才算正式版本。LINK 分钟级的速度使其成为最适合的冒烟闸门。

**代价**：
- LINK 基础设施必须稳定可用
- 冒烟套件要维护（反映融合正确性的最小用例集）

**可选方案**：
- 不做冒烟：~~融合错误流到下游平台浪费资源~~
- 冒烟跑 FPGA：~~慢；发现问题来不及~~
- 冒烟跑 EMU：~~慢且资源稀缺~~

**推翻条件**：LINK 不可用或变得比其他平台慢。

**相关**：01 · 1.5

## ADR-11：SDK / 平台软件 / 编译器走 CMC 分发

**状态**：Accepted

**上下文**：SDK、平台软件（运行时）、编译器是**平台无关的通用组件**，应该上游统一发布，不由本项目构建。

**决策**：这三类组件**全部从 CMC 制品仓库拉取**，本项目不再构建。业务软件的打包脚本与跨模块共享资源由 **smartci 仓库**维护。FPGA / LINK / EMU 对这三类组件的处理**一视同仁**（消除旧文档里 "FPGA SDK 独立 / LINK SDK 内嵌"的不对称）。

**代价**：
- 依赖 CMC 制品仓的可用性
- 版本对齐需要约束（基线 manifest 显式指定组件版本）

**可选方案**：
- 本项目重复构建：~~维护成本高，和上游脱节~~
- 不同平台用不同版本策略：~~造成文档对称性混乱，已被此 ADR 修正~~

**推翻条件**：CMC 停服或这三类组件的来源模型变化。

**相关**：00 · 2.3，01 · 1.1 / 2.3 / 2.4 / 3.3

## ADR-12：组件版本校验用 VerFingerPrint 日志打点

**状态**：Accepted

**上下文**：版本加载后需要验证"加载到 CPU 桩 / FPGA 的确实是期望版本"。传统做法是 `reg_read` 某个 version 寄存器，但多组件场景下要读多次、各组件的 version 寄存器位置还不一样。

**决策**：每个关键组件（SDK / 平台软件 / 业务软件 / CPU 桩 / 软调）在启动期 DEBUG 日志中**打印一条版本指纹**，格式：

```
VerFingerPrint[<组件>]: <指纹字符串>
```

示例：`VerFingerPrint[biz_sw]: 65010219930822401X`

Autotest 在版本加载阶段解析日志并对照 manifest。

**代价**：
- 需要各组件遵守日志格式（前期人为约定 + CI 校验）
- 日志量略增（可接受，DEBUG 级别本来就存在）

**可选方案**：
- `reg_read` 版本寄存器：~~多组件场景重复；地址可能变；无法统一解析~~
- 专门的 version 查询 RPC：~~工程量大，没有必要~~

**推翻条件**：某些组件无法改日志（如外购二进制），需要改用其他通道校验。

**相关**：00 · 4.3，01 · 2.10

## 何时该新增 ADR

出现以下情况时写一条 ADR：

1. 跨平台 / 跨模块的架构选型（影响超出单文件）
2. 业务规则的根本判定（如"精度是 bit 级"）
3. 外部依赖的引入或替换（如 "用 LAVA"）
4. 推翻之前的决策（标记旧 ADR 为 Superseded）

**不该写 ADR 的**：日常代码实现细节、某个 adapter 的内部组织、某个 fixture 怎么写。这些属于 06 软件架构或 09 用例开发指南。
