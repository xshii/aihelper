## 1. 现状与驱动

### 1.1 现状

当前 Autotest **只支持 FPGA 原型（上板）** 一条链路。其他平台的基础设施都已存在，但**没有统一接入 Autotest**：

| 平台 | 基础设施现状 | Autotest 是否接入 |
|---|---|---|
| **link**（功能仿真）| 已发布，可单独跑 | ❌ |
| **rtl2c** | 脚本 + 工具链已有 | ❌ |
| **fpga**（原型板）| 现有链路全打通 | ✅ |
| **emu**（硬件仿真器）| 自有 API + Python 接口 | ❌ |
| **eda** | EDA 环境已有 | ❌ |

### 1.2 驱动（为什么必须扩展）

只靠 fpga 做不到的事：

| 痛点 | 影响 |
|---|---|
| **FPGA 性能不准** | 性能用例无权威出口，回归不可信 |
| **FPGA 板级资源稀缺** | 排队严重、回归慢 |
| **FPGA 无法导出波形** | 失败定位只能靠 dump，低级时序问题看不到 |
| **FPGA 加载耗时长** | 早期开发自测成本高 |
| **FPGA 失败难定位** | 没有第二个平台交叉验证 |

业务上必须走多平台才能解这些痛点——不是"锦上添花"，是硬需求。

### 1.3 本次扩展不做什么

- **不搭建平台**（基础设施都已存在）
- **不替代 DV 团队的 RTL 验证**
- **不替代人工波形分析**（eda 只做交付，不做自动分析）

核心工作是"**把已有平台封装进 Autotest、让用例跨平台统一跑、让回归有统一编排**"。

## 2. 平台职责分工（业务视角）

每个平台在回归流程中承担独立职责，不是同一件事的五种实现。

| 平台 | 业务职责 | 为什么是它 |
|---|---|---|
| **link** | **准入闸门** + L0 冒烟 | 最快最便宜，卡住不合格版本省 FPGA 板级资源 |
| **rtl2c** | **波形辅助定位**（主回归并行路径）| 能导出波形（FPGA 无此能力），失败时的低级时序定位必需；顺带作为主回归第二条 RTL 实现交叉校验 |
| **fpga** | **功能主力主回归** + 长稳 | 真硬件速度、链路最完整 |
| **emu** | **性能权威出口** + **FPGA 失败诊断** | 硬件仿真器性能可信；支持 checkpoint / replay，调试效率高 |
| **eda** | **交付给人工波形分析** | Autotest 不在 EDA 执行用例，只打包上传 |

### 2.1 link ‖ rtl2c 并行的业务含义

两者并行不是冗余：

- **link 负责准入**（闸门，失败阻断 fpga）
- **rtl2c 负责波形**（失败时的定位手段）
- rtl2c 执行慢，link 失败时 rtl2c 的波形仍有价值（诊断用途）
- 两者同源于 RTL，若 rtl2c FAIL 大概率 fpga 也 FAIL（反之亦然），起**交叉校验**作用

### 2.2 emu 的双重角色

| 角色 | 触发 |
|---|---|
| 性能出口 | fpga 主回归通过后，对性能用例在 emu 上复测 |
| FPGA 失败诊断 | fpga 失败时，emu 和 eda 并行触发诊断 |

### 2.3 eda 只交付不执行

Autotest 不在 eda 中跑用例、不采集 eda 结果——只负责"编译 → 打包 → 上传到指定位置"，由人手动在 eda 环境做波形分析。

**两种触发方式：** 自动（fpga/rtl2c 失败时）+ 人工（按需）。

## 3. 功能需求

### FR-1 用例配置相通

同一份用例定义（pytest 函数 + markers + parametrize）可在任意可执行平台执行，**不因目标平台改写用例代码**。

### FR-2 判定标准相通

跨平台使用**一致的裁决标准**：

| 维度 | 标准 |
|---|---|
| 功能 | **bit 级一致**（见 02 · 第 7 章）|
| 性能 | 基线偏差阈值（见 03 · 3.4，性能以 emu 为准）|
| DFX | 联合判定（见 03 · 5.2 和 04）|

判定逻辑写一次、各平台复用，**不允许平台特例的裁决分支**。

### FR-3 跨平台 bit-exact 输出一致

**link / rtl2c / fpga / emu 同一份用例必须产出完全相同的输出**：

- 四者都是**同一 RTL 语义**的不同实现载体
- 仿真不是"算法参考"，而是"RTL 的软件等价实现"
- 任何跨平台差异都是 bug，必须定位
- 前提：同一基线下 `rtl_commit` + `gc_version` 一致（详见 01 · 7.2 和 03 · 2.2）

eda 不执行用例，不参与 bit-exact 校验。

### FR-4 出包差异化封装

各平台出包脚本和执行命令不同（见 01 各章），Autotest 侧需**屏蔽差异**：

- 上游（用例作者、编排器）不感知平台差异
- 差异封装在平台适配层（见 wiki 6 · 3 领域分层）

### FR-5 资源调度与排队

- 多 FPGA 板、多 emu 实例的分配与排队由**资源调度服务**承担（对标 LAVA，详见主文档 4.7.5）
- 调度对用例透明，pytest 侧只感知"拿到一个可用资源"

### FR-6 自动失败定位（bisect）

主回归失败时触发 `git bisect` 自动定位首坏 commit，缩小排查范围。

### FR-7 自动 EDA 打包

fpga / rtl2c 失败时自动触发 EDA 打包上传，缩短人工分析启动时间。

### FR-8 EMU checkpoint / replay

EMU 支持中间状态保存 + 断点续跑；调试长推理时先跑到嫌疑阶段再反复试。FPGA 是否支持**待确认**。

## 4. 非功能需求

### NFR-1 回归总耗时

| 阶段 | 期望时长 | 备注 |
|---|---|---|
| 准入（link ‖ rtl2c） | 分钟 ~ 十分钟级 | link 是闸门，rtl2c 可后台继续 |
| 主回归（fpga） | 小时级 | 真硬件速度 |
| 性能（emu） | 小时 ~ 数小时级 | 仅跑性能用例子集 |
| 诊断（emu 重跑 + eda 打包）| 失败时触发 | 不阻塞主回归结束 |

### NFR-2 资源约束

- FPGA 板数稀缺 → 资源调度必须做
- emu 机位稀缺 → 同上
- link / rtl2c 机器资源弹性 → 不易成为瓶颈

### NFR-3 扩展性

新增一个执行平台时，**核心业务逻辑（精度比数、裁决、编排）零修改**——只动平台适配层。具体实现约束见 wiki 6。

### NFR-4 可观测性

所有平台产出**统一结构的报告**（见 [子 Wiki 7](./07_子wiki_验证报告模板.md)），`platform` 字段区分。失败能追溯到用例 ID + 基线 + RTL commit + GC 版本。

### NFR-5 可追溯性

每个报告带 `baseline_id` / `rtl_commit` / `gc_version` / `prototype_version` / VerFingerPrint 五元组，失败可回到具体版本复现。

### NFR-6 pytest 主体约束

Autotest 主体是 **pytest**，所有平台适配必须以 pytest fixture / markers / parametrize / plugin 形式融入，不另造 runner。

## 5. 用户角色与使用场景

| 角色 | 核心需求 | 关联产物 |
|---|---|---|
| 验证开发 | 写一次用例，多平台复用 | pytest 用例 + markers |
| 验证执行 | 按套件或标签一键跑回归；看报告；失败快速定位 | 套件配置 + 报告（wiki 7）|
| 芯片 / 软件开发 | 读报告、bisect 结果、首发散阶段 | 报告 + 辅助分析 |
| TL / PM | 看回归趋势、性能 drift、Top Bug | 趋势报告（wiki 7 · 4）|
| 值班 | 失败自动 triage、自动打包 eda | bisect + 自动打包 + 通知 |

### 5.1 典型使用场景

**场景 A：日常回归**

回归触发 → link 准入（失败阻断）→ fpga 主回归 + rtl2c 并行 → emu 跑性能用例 → 归档。

**场景 B：功能用例失败**

fpga 某用例功能 FAIL → Autotest 同时触发 emu 诊断 + eda 自动打包 → rtl2c 侧如有对应波形可查 → 工程师用 emu 的 checkpoint / replay 定位首发散阶段。

**场景 C：性能回归**

fpga 主回归通过 → emu 跑性能用例 → 若某指标超 +10% 判 FAIL；若连续 N 轮 drift 进入预警（见 wiki 7 · 4）。

**场景 D：跨平台不一致**

同一用例 link / rtl2c / fpga / emu 输出不一致 → bit 比对找首发散 → 定位差异在哪一层（算法 / 时序 / 综合 / emu 工具链），详见 6.3。

## 6. 验收标准

按阶段验收。**每阶段验收通过后才进入下一阶段**。

### 阶段 0：现有 fpga 链路重构

- 验收项：已有 fpga 用例通过率**不下降**
- 产物：fpga 链路通过适配层调用，核心业务层零变化

### 阶段 1：link ‖ rtl2c 接入

- 验收项：link 作为准入闸门可用；rtl2c 并行波形导出可用
- 跨平台校验：L0 用例在 link / fpga 上 bit-exact 一致

### 阶段 2：emu 接入

- 验收项：emu 性能出口可用；emu 诊断回归可用；checkpoint / replay 演示通过
- 跨平台校验：同一基线的回归子集在 fpga / emu bit-exact 一致（性能值允许不同）

### 阶段 3：eda 自动打包

- 验收项：fpga / rtl2c 失败自动触发 eda 打包上传；打包产物可被人工 EDA 环境加载

### 阶段 4：整体编排 + 资源调度

- 验收项：整轮回归一键触发；资源调度服务对接（LAVA 或等价）；pytest-xdist 多板并行可用
- 可观测性：所有平台报告结构统一（见 wiki 7）

### 阶段 5：跨平台一致性闭环

- 验收项：差异自动告警；失败自动 bisect + EDA 打包 + emu 重跑三线并行
- 趋势报告可产出

## 7. 开放问题

需要与相关团队对齐才能落地的事项：

| # | 问题 | 谁来对齐 |
|---|---|---|
| 1 | **link 接口**：命令行接口、输入 / 输出格式、dump 机制具体定义 | 仿真发布团队 |
| 2 | **rtl2c 脚本约定**：入参、输出路径、返回码；波形采集开关怎么传 | rtl2c 维护方 |
| 3 | **EMU Python API 覆盖范围**：espi 控制是否覆盖所有 fpga 上的 CPU 桩场景；TESGINE 灌数据的格式约束 | emu 工具链团队 |
| 4 | **EMU 性能标定**：emu 壁钟时间如何与"硅后真实性能"预测对齐；基线更新频率 | emu + 架构 |
| 5 | **checkpoint / replay 覆盖范围**：EMU 已支持需确认覆盖面；FPGA 是否能做待硬件侧确认 | emu 工具链团队 + 硬件 / 板级团队 |
| 6 | **EDA 打包触发粒度**：任一用例失败触发还是套件失败触发；上传目标路径 | EDA 环境维护 + 产品 |
| 7 | **跨平台差异定位责任**：link vs rtl2c / fpga / emu 差异由谁定位 | 架构委员会 |
| 8 | **基线一致性保证**：各平台 RTL commit + GC 版本同步机制 | CI / 版本管理 |
| 9 | **资源调度方案选型**：直接引入 LAVA vs 自建最小化调度服务 | 架构 + 基础设施 |
| 10 | **onebox 是否有自动化上传下载接口**（GOLDEN 归档，见 03 · 2.2.4）| 基础设施 |
