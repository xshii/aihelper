## 1. 比数模式概述

Autotest 的比数能力分为两种模式，二者互斥，不能同时启用：

| 模式 | 定义 | 业务软件 | 适用 |
|---|---|---|---|
| **端到端比数模式** | **使用原始业务软件**，不做任何阶段性导出，只看最终结果 | 原始业务软件（不修改） | 常规高频执行、性能基线测量 |
| **阶段性比数模式** | **业务软件配合导出部分中间张量**并比对 | 业务软件三个版本之一（见第 4 章） | 精度定位、首用例准入 |

### 1.1 两种模式的本质区别

- **端到端比数模式**：业务软件按生产行为运行，**不做任何额外 dump**，不涉及 ID 规划、不涉及内存特殊布局。对性能无任何扰动。
- **阶段性比数模式**：业务软件在设计阶段预置了**ID 体系 + 内存规划**，在推理过程中按阶段导出中间张量，由接口 FPGA 或 Autotest 比对工具完成比数。根据业务软件版本不同，对性能的扰动程度、覆盖范围、资源占用都不同。

### 1.2 关键认知：阶段性比数需要业务软件配合

阶段性比数**不是运行时动态注入的 dump**，而是**业务软件架构设计阶段就预置好的能力**：

- 要导出的中间张量 ID 要预先规划
- 内存布局要配合 ID 使用
- 触发开关与 Autotest 接口要在业务软件中实现

这是跟"端到端模式（用原始软件）"最大的区别——**切换到阶段性比数，不是改 Autotest 配置那么简单，而是要用专门的业务软件版本**。

### 1.3 不是"全量逐层 dump"

阶段性比数**不采用逐层全量 dump**作为默认策略：

- 全量 dump 数据量巨大，传输回读耗时不可接受
- 大多数精度问题可以在"阶段性"粒度定位，不必全量
- 业务软件版本的设计也是基于"部分采样"思路，不是每层都抓

## 2. 业界参考与设计思路

### 2.1 业界做法

| 框架 / 厂商 | 能力 | 可借鉴点 |
|---|---|---|
| NVIDIA TensorRT | `--dumpOutput` / profiler hook | 分层 hook 架构 |
| 华为 CANN (msaccucmp/msame) | 按算子 dump + 离线比对工具 | dump 格式化 + 标准比对工具 |
| PyTorch / ONNX Runtime | reference runtime 生成 golden | 业界参考；**本项目不用**，bit 级一致要求决定 GOLDEN 必须来自 **GC 硬件计算功能仿真**（见子 Wiki 3 · 2.2）|
| MLIR 生态 | IR 层断点、转储 | 编译器与运行时联动 |
| MLPerf | 精度准入阈值规范 | 仅作业界对照；本项目不用容忍度，裁决 = bit 级一致（见第 7 章）|

**业界共识：** 采样粒度**按需可调**，不强求全量 dump。Autotest 的阶段性比数思路与此一致。

### 2.2 Autotest 的设计原则

1. **对现有主流程零侵入**：比数通过配置开关启用，不影响常规回归
2. **两种模式分离**：端到端比数与阶段性比数分别有独立实现路径，不互相干扰
3. **标准化数据格式**：dump 数据用业界通用格式（npy + 元数据 json）
4. **多层协作**：Autotest 平台 + CPU 桩 + 运行时三层共同实现

### 2.3 三层协作架构

```
┌─────────────────────────────────────────────┐
│  Autotest 平台层                             │
│  - 策略决策（端到端 vs 阶段性）              │
│  - 模式自动切换                              │
│  - 执行路径自动选择（标准 / 备选，见 4.1）    │
│  - 数据收集 + 比对工具调用 + 报告            │
└───────────────┬─────────────────────────────┘
                │
┌───────────────▼─────────────────────────────┐
│  CPU 桩侧（由 Autotest 编排）                │
│  - 标准路径：加载到接口 FPGA、配定时器       │
│  - 备选路径：软调直读写被测 FPGA、做 dump    │
└───────────────┬─────────────────────────────┘
                │
┌───────────────▼─────────────────────────────┐
│  平台软件 / 被测 FPGA 侧                     │
│  - 阶段结束点 hook                           │
│  - 按配置输出端到端或阶段性张量              │
└─────────────────────────────────────────────┘
```

## 3. 比数策略：精度用例 vs 性能用例

两类用例的比数策略完全不同。核心差异：默认模式不同，切换触发方式不同。

### 3.1 精度用例策略

精度类用例关心**结果正确性**，一上来就要能定位精度偏差，因此：

```
          精度用例集
              │
              ▼
        识别首用例
     （CPU 简化、无硬件优化的版本）
              │
              ▼
   首用例 → 阶段性比数模式（默认）
              │
           通过?
              │
         ┌────┴────┐
        是        否
         │         │
         ▼         ▼
  【自动切换】    定位分析
   端到端比数     （看首发散阶段）
   模式
   （跑正常规格
    用例集）
```

**关键规则：**

- **首用例**先用阶段性比数跑通，作为精度准入 —— 见第 4 章
- 首用例通过 → **Autotest 自动切换**到端到端比数模式 —— 不需要人工配置
- 首用例失败 → 进入定位流程（找首发散阶段）
- 正常规格用例走端到端比数模式，效率高

### 3.2 性能用例策略

性能类用例关心**吞吐与时延**，阶段性 dump 会干扰计时，因此默认端到端：

```
        性能用例
           │
           ▼
    端到端比数模式（默认，优先）
           │
        通过?
           │
      ┌────┴────┐
     是        否
      │         │
      ▼         ▼
    归档    触发阶段性比数模式
             （自动或手动）
             │
             ▼
            定位
```

**关键规则：**

- **端到端比数默认优先**，保证性能测量不被扰动
- 失败时触发阶段性比数重跑 —— **自动与手动两种触发方式都支持**：
  - **自动**：在 CI 或回归 pipeline 中配置"失败自动回落"，端到端失败后自动用阶段性比数重跑
  - **手动**：失败后由工程师改脚本/配置触发阶段性比数
- 阶段性比数重跑**用于定位**，不作为性能判定依据（计时失真）

### 3.3 两种策略对照

| 维度 | 精度用例 | 性能用例 |
|---|---|---|
| 默认模式 | 阶段性比数（首用例）| 端到端比数 |
| 正常规格用例模式 | 端到端比数（首用例通过后）| 端到端比数 |
| 模式切换触发 | **自动**（首用例通过后）| **自动 + 手动**（端到端失败时）|
| 阶段性比数作用 | 准入 + 精度定位 | 失败后的定位辅助 |
| 性能数据 | 不作为主要关注点 | 端到端比数产出 |

### 3.4 业务软件版本是独立维度

上面的策略决定了**什么时候用阶段性比数**。而"**阶段性比数用哪个业务软件版本**"是**独立的第三维**——业务软件三个版本（初级/进阶/商用，见第 4 章）**可以混用，按需选择**，不与用例类型严格绑定。

三维关系总览：

| 维度 | 选项 | 谁决定 |
|---|---|---|
| 用例类型 | 精度 / 性能 | 用例作者 |
| 比数模式 | 端到端 / 阶段性 | 用例作者 + Autotest 策略（首用例通过后自动切换等）|
| **业务软件版本**（仅阶段性模式下需要选） | 初级 / 进阶 / 商用 | 用例作者 |

> **执行路径（标准 / 备选）不是这里的维度**——路径由运行时环境的"消息通道是否通"决定，Autotest 自动选，不是用例配置项（见 4.1）。

典型组合示例（仅参考，实际按需选择）：

| 场景 | 用例类型 | 比数模式 | 业务软件版本 |
|---|---|---|---|
| 首用例准入 | 精度（首用例） | 阶段性 | 初级（每结果导出） |
| 精度深度定位 | 精度 | 阶段性 | 初级 |
| 性能基线测量 | 性能 | 端到端 | —（原始软件） |
| 性能失败回落定位 | 性能 | 阶段性 | 进阶（算子轮换，减少扰动） |
| 商用场景真实验证 | 性能 | 阶段性 | 商用（DMA 流量真实） |

### 3.5 用例配置示例

**精度用例集配置：**

```yaml
suite: cv_resnet50_accuracy
markers: [accuracy]                  # pytest markers
首用例: cv_resnet50_cpu_baseline     # 仅此一项必填；其余规则框架固化
业务软件版本: 初级                   # 阶段性比数期间用哪个版本
cases:
  - cv_resnet50_cpu_baseline         # 首用例
  - cv_resnet50_normal_001
  - cv_resnet50_normal_002
  ...
```

> 框架固化规则：首用例走 `stage_compare` → 通过后自动切 `end_to_end`（见 3.1 / 5.5）。

**性能用例集配置：**

```yaml
suite: cv_resnet50_perf
markers: [perf]                      # 性能用例默认 end_to_end
失败回落:
  mode: auto                         # auto | manual | off（默认 auto）
  业务软件版本: 进阶                 # 回落后用哪个业务软件版本
cases:
  - cv_resnet50_perf_bs1
  - cv_resnet50_perf_bs8
  ...
```

> 框架固化规则：性能用例默认 `end_to_end`；失败回落目标固定为 `stage_compare`（见 3.2）。

## 4. 业务软件配合阶段性比数的三个版本

阶段性比数需要**业务软件配合**。根据内存策略、导出范围、对性能扰动程度的不同，业务软件分为三个演进版本，可以混用、按需选择。

### 4.1 两种比数执行路径

阶段性比数有两条执行路径，**标准路径**由接口 FPGA 在硬件层面完成 bit 精准比对，**备选路径**由 CPU 桩 dump + 独立仓库管理的 compare 比数模块在主机侧完成。

#### 4.1.1 标准路径：接口 FPGA 硬件比对

被测 FPGA 与接口 FPGA 配合完成**硬件级 bit 精准比对**，Autotest 不参与具体数据比对：

```
          被测 FPGA                        接口 FPGA
         ┌─────────┐                      ┌───────────┐
         │ 业务推理 │                      │  GOLDEN   │
         │         │                      │  存储区    │
         │ 中间张量 │                      │ (按 ID)   │
         │ (DDR)   │                      └─────┬─────┘
         └────┬────┘                            │
              │  ① 发 ID + DDR 地址              │
              ├─────────────────────────────────►
              │                                 │
              │                          ② 按 ID 查表 → GOLDEN 位置 + 比对大小
              │                                 │
              │  ③ 接口 FPGA 读被测 FPGA DDR     │
              │◄────────────────────────────────┤
              │                                 │
              │                          ④ bit 级精准比对
              │                                 │
              │                                 ▼
              │                         PASS / FAIL（寄存器 / 中断上报）
```

**关键点：**

- 流程：被测 FPGA 发 **ID + DDR 地址** → 接口 FPGA 按 ID 确认 GOLDEN 存放位置与比对大小 → 读取被测 FPGA DDR 数据 → bit 精准比对
- 比对全在硬件侧完成，**对业务推理几乎零扰动**
- 裁决结果仅 PASS / FAIL 两态（硬件天然如此），辅助分析不在此路径
- GOLDEN 必须预先烧入接口 FPGA 的存储区（有容量上限）

**CPU 桩 ↔ 被测 FPGA 的所有交互**（配置下发 / 数据加载 / 激励触发 / 状态查询 / 结果读取）都走接口 FPGA 消息通道：CPU 桩把权重 / 激励 / GOLDEN 加载到接口 FPGA，接口 FPGA 按 CPU 桩配置的**定时器参数**下发。整个流程由 Autotest 编排。

**约束（沿用接口 FPGA 的能力边界）：**

- **ID 体系**必须在业务软件设计阶段预先规划——每个要比数的中间张量都要有唯一 ID
- **内存布局**要配合 ID，被测 FPGA 按 ID 导出数据到特定地址，接口 FPGA 按同样 ID 取 GOLDEN 比对
- 接口 FPGA **不做复杂张量处理**（不做 shape 推断、不做格式转换）——所有前置工作在业务软件中完成

**挑战**：在软件设计阶段就要把 ID 和内存规划做好，这对业务软件架构提出较高要求。4.2~4.4 的三个业务软件版本的差异主要体现在如何优化这部分实现。

#### 4.1.2 备选路径：CPU 桩 dump + compare 模块

当标准路径不可用（接口 FPGA 资源受限、GOLDEN 超限、调试场景需要留原始数据、非 FPGA 平台等）时走备选路径：

```
          被测 FPGA              CPU 桩                       主机
         ┌─────────┐           ┌─────────────┐          ┌──────────────┐
         │ 全局变量 │◄──────────┤ ① 读全局变量 │          │              │
         │（指向    │           │   → 结果位置 │          │              │
         │  结果区）│           │             │          │              │
         │         │           │ ② 读对应    │          │              │
         │ 结果数据 │◄──────────┤   DDR 内容  │          │              │
         │ (DDR)   │           │   做内存 dump│          │              │
         └─────────┘           │             │          │              │
                               │ ③ dump 文件 │          │              │
                               │   回传主机  ├─────────►│              │
                               └─────────────┘          │ ④ compare    │
                                                        │   比数模块   │
                                                        │   bit 比对   │
                                                        │   + 辅助分析 │
                                                        └──────┬───────┘
                                                               │
                                                               ▼
                                                       PASS / FAIL
                                                       + 综合报告（第 7 章）
```

**关键点：**

- 流程：CPU 桩**读被测 FPGA 全局变量**确认结果存放位置 → **读对应 DDR 内容做内存 dump** → dump 回传主机 → **compare 比数模块**做 bit 比对 + 辅助分析
- **compare 比数模块独立仓库管理**，便于跨平台 / 跨场景复用（link / rtl2c / fpga / emu 都能调）
- 无需接口 FPGA 预存 GOLDEN，主机侧直接和本地 GOLDEN 比对
- FAIL 时直接产出辅助分析报告（分 block / LCS 偏移 / 位域 / QSNR，见第 7 章）

**CPU 桩 ↔ 被测 FPGA 的所有交互**（配置下发 / 数据加载 / 激励触发 / 全局变量改写 / 结果读取）统一走**软调提供的底软接口**（读写被测 FPGA 内存）。典型差异：激励触发改成**改写全局变量 → 自触发模式**。整个流程由 Autotest 编排。

**适用：**

- 接口 FPGA 资源受限 / GOLDEN 放不下
- 调试场景需要保留原始 dump 数据做事后分析
- 非 FPGA 平台（link / rtl2c / emu）
- 标准路径裁决 FAIL 后的辅助定位

**代价：**

- CPU 桩读 DDR 有扰动，不适合性能基线测量
- dump 回传有传输开销
- 比对时延高于硬件路径

#### 4.1.3 两条路径对照

| 维度 | 标准路径（接口 FPGA）| 备选路径（CPU 桩 + compare）|
|---|---|---|
| 比对位置 | 硬件（接口 FPGA 内） | 软件（主机侧 compare 模块） |
| GOLDEN 存放 | 接口 FPGA 存储区（有容量上限）| 主机文件系统 |
| CPU 桩 ↔ 被测 FPGA 交互 | 经接口 FPGA 消息通道下发（数据、配置、激励、状态查询）| 通过软调底软接口直接读写被测 FPGA 内存 / 全局变量 |
| 激励触发形式 | 接口 FPGA 按 CPU 桩配置的定时器定时发 | 改写被测 FPGA 全局变量为自触发模式 |
| 性能扰动 | 几乎无 | 有（CPU 桩读写 DDR + dump 回传）|
| 裁决输出 | PASS / FAIL（硬件只能出这两态）| PASS / FAIL + 辅助分析报告 |
| 跨平台复用 | 仅 FPGA（依赖接口 FPGA） | 所有可执行平台 |
| 模块归属 | 业务软件 + 接口 FPGA 硬件逻辑 | **compare 比数模块 = 独立仓库**；CPU 桩读写依赖**软调**提供的底软接口 |

**适用原则（两条路径的分水岭）：** **凡 CPU 桩 ↔ 被测 FPGA 的交互都存在两条路径**——数据加载、配置下发、用例选择、激励触发、状态查询、结果读取，都是如此。选择哪条取决于 **CPU 桩 ↔ 接口 FPGA ↔ 被测 FPGA 的消息下发通道是否通**：通则走标准路径；不通（路径未就绪 / 接口 FPGA 资源受限 / 调试态）则退到备选路径，走软调直写。CPU 桩侧的所有流程都由 **Autotest 编排**。

**关系：** 两条路径**互补，不互斥**：

- 标准路径负责"快速 bit 级 PASS/FAIL 裁决"
- 备选路径负责"FAIL 后的辅助定位 + 非 FPGA 平台比数 + 调试场景的原始数据留存"
- Autotest 可在同一用例中先走标准路径裁决，FAIL 时自动触发备选路径做辅助分析

**以下 4.2~4.4 讲的业务软件三个版本（初级 / 进阶 / 商用），都是针对标准路径的设计差异**——备选路径不涉及这种业务软件演进。

### 4.2 初级版本

**特点：**

- **内存非复用**：每个中间张量独立分配存储空间
- **每结果导出**：所有配置了 ID 的中间张量都导出比对

**优点：**
- 实现最简单，ID 规划相对直接
- 覆盖完整，方便定位任何位置的精度问题

**缺点：**
- 内存占用大
- 对性能扰动大（每次都导出，DMA 持续占用）

**适用场景：**
- 功能用例
- 首用例准入
- 精度深度定位

### 4.3 进阶版本

**特点：**

- **内存复用**：中间张量共享存储空间
- **每轮选择不同算子进行结果导出**：多轮执行，每轮只导出一部分算子的结果，多轮拼起来看全貌

**优点：**
- 内存占用可控
- 每一轮的扰动降低（单轮只抓一部分，减少 DMA 冲击）

**缺点：**
- 实现复杂度提高（内存复用要仔细规划不冲突）
- 需要多轮执行才能拿到完整数据

**适用场景：**
- 性能用例失败后的回落定位
- 内存受限环境下的阶段性比数

### 4.4 商用版本

**特点：**

- **内存复用** + **考虑 DMA 流量**
- 导出真实商用场景的数据
- **设计上完全不影响性能用例的测量**

**优点：**
- 可用于商用生产环境
- 性能用例开启它也不会导致计时失真
- 数据最贴近真实业务行为

**缺点：**
- 设计和实现最复杂
- ID 和内存规划需要在业务软件架构初期就完成

**适用场景：**
- 商用场景真实验证
- 对性能零扰动要求下的阶段性比数
- 长期稳定运行监控

### 4.5 三版本对照

| 维度 | 初级版本 | 进阶版本 | 商用版本 |
|---|---|---|---|
| 内存策略 | 非复用 | 复用 | 复用 + DMA 流量规划 |
| 导出策略 | 每结果导出 | 每轮选不同算子 | 真实商用场景 |
| 对性能的扰动 | 大 | 中 | 几乎无 |
| 单次执行覆盖 | 完整 | 部分（多轮拼全） | 按配置 |
| 实现复杂度 | 低 | 中 | 高 |
| 典型适用 | 功能用例、首用例、精度定位 | 性能失败回落 | 商用场景验证 |

### 4.6 版本演进关系

三个版本是**演进关系**，同时也可混用：

```
初级版本     ──► 进阶版本     ──► 商用版本
(起步)         (内存优化)      (性能无扰动)
   │              │                │
   ├──────────────┼────────────────┤
   ▼              ▼                ▼
  可以同时存在，按用例需要选择
```

Autotest 在用例配置中指定使用哪个版本，执行时调用对应版本的业务软件加载。

### 4.7 业务软件的设计阶段要求

标准路径下，以下设计工作必须在业务软件开发阶段完成（ID 体系 / 内存布局的约束已在 **4.1.1** 交代，这里只补三个版本的差异要点）：

- **ID 体系设计**：每个可能需要比数的中间张量分配唯一 ID，形成全局 ID 表（4.1.1 前置说明）
- **内存规划**：按版本策略（独立 / 复用）规划内存布局；**复用版本（进阶 / 商用）要保证时序不冲突**
- **触发开关**：业务软件支持按配置开关指定导出的 ID 集合
- **接口对齐**：被测 FPGA 导出的数据格式、地址、节拍与接口 FPGA 读取 GOLDEN 的方式对齐

这部分是业务软件架构的**硬约束**，不是事后添加的可选功能。备选路径不涉及这些约束——它通过软调直读 DDR 做 dump，由主机侧 compare 模块比对。

## 5. 首用例机制

### 5.1 首用例是什么

**首用例**：一批精度用例的打头阵用例，是**实现层面的"简化版"**，核心特征：

- **CPU 简化实现**：算子走 CPU 路径的简化实现，不走硬件加速通路；golden 容易算、实现 bug 少
- **不使用高级硬件优化特性**（如 DMA 双缓、tiling、融合算子等），减少硬件相关的 bug 源
- **对端到端时延无要求**：只验证正确性，不关心性能

换言之，首用例是"先把精度链路跑通"的参考实现；它不是正常规格用例的简单缩小，而是**换了一种更简单、更可信的执行路径**。后续正常规格用例在此基础上逐步打开硬件优化特性，一步一步把失败范围缩小到具体的优化点。

### 5.2 为什么需要首用例

| 目的 | 说明 |
|---|---|
| **排除硬件优化干扰** | CPU 简化路径 + 不开硬件优化特性，失败只可能来自基础算子 / 数据通路，bug 源空间大幅收敛 |
| **准入门槛** | 首用例失败直接阻断，不浪费正常规格用例的硬件资源 |
| **定位锚点** | 先在 CPU 简化路径上跑通阶段性比数拿到可信参照，再逐步打开硬件优化，失败点自然落在打开的那一项 |

### 5.3 首用例的选择与维护

- 首用例**不是正常规格用例的简单缩小版**，而是同一套精度链路的"CPU 简化 + 无硬件优化"版本，需要精心设计：
  - 算子覆盖要具代表性（覆盖正常规格用例中的关键算子类别）
  - 阶段划分要与正常规格一致（这样首发散阶段的结论可以直接迁移）
  - **关闭所有硬件优化特性**（DMA 双缓 / tiling / 算子融合 / 流水等），只保留基础执行路径
  - 精度裁决与常规用例一样：**bit 级一致**（不存在"阈值稍严"的概念）；CPU 简化实现只是让失败的 bug 源空间更收敛，不改变裁决标准
- 首用例与正常规格用例**在用例库中分别标记**
- 每个精度用例集应至少有 1 个首用例

### 5.4 首用例的阶段性比数配置

首用例默认启用阶段性比数，并明确比数的采样点：

```yaml
case: cv_resnet50_cpu_baseline
markers: [首用例, accuracy]
model: resnet50
execution_mode: cpu_simplified       # cpu_simplified | hw_optimized
                                     # cpu_simplified 隐含关闭所有硬件优化 + 不看端到端时延
stages:
  - {name: backbone_out, end_layer: layer4.2.relu}
  - {name: neck_out,     end_layer: fpn.out}
  - {name: head_out,     end_layer: classifier}
# 精准比数裁决 = bit 级一致（固定），辅助分析 FAIL 时自动启用（见第 7 章）
```

### 5.5 首用例通过后的自动切换

Autotest 在首用例通过后**自动**把比数模式切换为端到端：

```
首用例跑完
   │
   ▼
Autotest 检查结果
   │
通过?
   │
┌──┴──┐
是    否
│     │
▼     ▼
自动  进入阶段性比数定位流程
切换  （参见 7.12 首发散阶段定位）
│
▼
套件内后续用例用端到端比数模式
```

切换逻辑由 Autotest 平台层决策，**不需要用户在配置中手动切换**。

## 6. Dump 机制设计

### 6.1 端到端比数不涉及 dump

**端到端比数模式使用原始业务软件，不做任何 dump**：

```
输入
 │
 ▼
[推理全过程]（原始业务软件，不修改）
 │
 ▼
最终输出 ──► 业务软件自然输出，与 golden 比对
```

- 对推理零扰动
- 不涉及 ID 规划、不涉及特殊内存布局
- 最终输出比对通过常规结果读取路径（见子 Wiki 3 · 第 4 章）

**本章节以下内容均针对阶段性比数。**

### 6.2 阶段性比数的 dump

阶段性比数由**业务软件配合**完成（三个版本见第 4 章），两条执行路径见 **4.1**：

- **标准路径**：dump 不落到主机，数据直接由被测 FPGA → 接口 FPGA 完成 bit 比对，以下 ID 比对机制图是此路径
- **备选路径**：CPU 桩读被测 FPGA 全局变量定位结果区，做内存 dump 回传主机，由独立仓库的 compare 模块比对（见 8.4）

以下以**标准路径**为主讲：

```
输入
 │
 ▼
[推理过程，业务软件按预先规划的 ID 导出部分中间张量]
 │      │
 │      ├── ID=001 → 被测 FPGA 导出 → 接口 FPGA 按 ID 取 GOLDEN 比对
 │      ├── ID=002 → ...
 │      └── ...
 │
 ▼
最终输出
```

**关键：** dump 不是运行时动态决定的，而是业务软件设计阶段确定的 ID 列表和内存布局。Autotest 在运行时指定"用哪个业务软件版本、开启哪组 ID"，不直接控制 dump 哪一层。

**关键：阶段划分由配置指定，不是每层都抓**。常见划分：

- 按网络结构（backbone / neck / head）
- 按算子簇（相同类型算子聚集区）
- 自定义（调试场景可指定层列表）

### 6.3 阶段划分的粒度选择

| 粒度 | 阶段数 | 适用 |
|---|---|---|
| 粗 | 2~4 个 | 首用例准入、快速定位 |
| 中 | 5~10 个 | 复杂模型的一般定位 |
| 细（接近逐层）| 10+ | 已知嫌疑区间的深度定位 |

粗粒度默认；只在定位到嫌疑区间后才开细粒度。**不建议直接上逐层全量 dump**。

### 6.4 触发方式

| 方式 | 描述 | 适用 |
|---|---|---|
| 全局开关触发 | 用例开始到结束全程按配置 dump | 阶段性比数常规用法 |
| 条件触发 | 输出异常（NaN / Inf / 超阈值）时才 dump | 稀疏错误场景 |
| 断点触发 | 指定阶段执行到时 dump 并可暂停 | 深度调试 |

### 6.5 数据回读通道

| 通道 | 适用数据量 | 速度 |
|---|---|---|
| 串口回显 | 小（如单张量 summary） | 慢 |
| 文件下载 | 中等（阶段性张量） | 中 |
| RDO 批量传输 | 大（多阶段批量） | 快，推荐 |

## 7. 精准比数工具

### 7.1 比对总原则

**精准比数的唯一通过标准是 bit 级完全一致**——任何一位不同即判 FAIL，不存在"接近就算过"。

这是由 AI 硬件验证的本质决定的：

- 跨平台（link / rtl2c / fpga / emu）被视为同一 RTL 语义的不同实现载体，输出必须完全相同
- 接口 FPGA 的硬件比对通道本身就是按 bit 比对 ID 对应的数据与 GOLDEN
- 任何非 bit-exact 差异都视为 bug，不允许用"容忍度"绕过

### 7.2 为什么还需要辅助分析手段

bit 级比对的结果只有 PASS / FAIL 两态，失败时提供的信息有限（只知道"不一致"，不知道"怎么不一致、差在哪、差多少"）。为了**提升问题定位效率**，精准比数工具在 FAIL 时自动输出多种辅助分析视角，帮助快速判断差异性质与可能根因。

**重要澄清：辅助分析的结果不改变裁决**——即使 QSNR 指标极高、只差一个 bit，依然判 FAIL。辅助分析只用于**定位**，不用于**判定**。

### 7.3 辅助分析手段总览

比对失败后，精准比数工具按需提供以下分析视角：

| 分析手段 | 作用 | 典型用途 |
|---|---|---|
| 分 block 比对 | 把张量切成若干 block，定位差异集中在哪个 block | 判断差异是全局扩散还是局部集中 |
| 偏移比对 | 基于最长公共子串自动发现 actual 与 golden 之间的错位量 | 判断是否是内存寻址/布局 bug 而非数值 bug |
| 浮点位域比对 | 分别比对指数位、符号位、尾数位 | 判断数值差异的性质（量级 vs 方向 vs 精度）|
| QSNR 比对 | 转为 double 后计算 QSNR | 量化差异规模，辅助判断严重程度 |

### 7.4 分 block 比对

**原理：** 把张量按固定大小切成若干 block，对每个 block 独立比对，输出"差异分布图"。

**输出内容：**

- 每个 block 的 PASS/FAIL 标记
- 差异 block 数 / 总 block 数
- 差异 block 的位置分布（可视化热图）

**典型结论模式：**

| 差异分布 | 可能指向 |
|---|---|
| 全部 block 都差 | 上游污染（输入或前置层已错）；数值格式整体偏差 |
| 差异集中在某一区域 | 特定算子 / 特定内存块问题 |
| 差异呈规律性分布（如每 N 个 block 一个错）| DMA 对齐 / 流水线节拍 bug |
| 仅首尾 block 差 | 边界处理 bug（padding、对齐处理）|

**可调参数**（默认启用，仅需覆盖默认值时写）：`block_size`（默认 64）—— 每个 block 的元素数。

### 7.5 偏移比对

#### 7.5.1 原理

当 bit 级比对整体失败、且怀疑是地址或布局错位（而非数值错误）时，不能靠"试几个固定偏移量"——真实错位量未知、可能是任意值。

偏移比对采用**最长公共子串（Longest Common Substring, LCS）**思路：把 actual 和 golden 都看作 bit 序列，寻找二者之间**最长的连续 bit-exact 匹配段**，由该匹配段自然反推错位量。

#### 7.5.2 算法流程

```
输入：actual[0..N-1]、golden[0..M-1]（按线性展开的元素序列）
输出：最长连续匹配段 {i, j, len} 及对应偏移量

1. 在 actual 和 golden 上执行 LCS（连续子串版本，不是最长公共子序列）
2. 找到最长 [i, j, len]，即 actual[i : i+len] 与 golden[j : j+len] 逐 bit 相等
3. 反推偏移量 offset = j - i
4. 可选：继续寻找次长、第三长的匹配段，判断是否多段错位
```

**实现层面的考量：**

- 朴素 LCS 复杂度 O(N·M)，对大张量不可行，实际用**后缀数组 / 后缀自动机 / rolling hash** 加速
- 可采用**分块 hash 预筛 → 候选区域精确比对**的两阶段方案
- float 数据按 bit pattern 匹配，而非数值比较
- 最小有效匹配长度应设阈值（如 ≥ 128 元素），避免被随机短匹配误导

#### 7.5.3 输出内容

| 输出项 | 含义 |
|---|---|
| 最长匹配段长度 | 连续 bit-exact 一致的元素数 |
| 匹配段在 actual 中起始位置 | `i` |
| 匹配段在 golden 中起始位置 | `j` |
| 反推偏移量 | `offset = j - i`（正负均可）|
| 匹配覆盖率 | 最长匹配段长度 / 总长度 |
| 次长匹配段 | 存在多段错位时提供 |

#### 7.5.4 判定辅助

| 最长匹配段结果 | 可能指向 |
|---|---|
| 匹配段长度 ≈ 总长度，offset ≠ 0 | 高置信度：**整体地址/布局错位**，优先查 DMA 起址、padding、layout 转换 |
| 存在多段较长匹配，offset 各不相同 | **分段错位**，可能是多次 DMA 传输中某几次错位 |
| 最长匹配段远短于总长度 | 不是单纯错位，可能叠加数值差异，需结合其他分析 |
| 匹配段长度接近于 0 或仅随机短匹配 | **数据完全不相关**，通常是输入/golden 对不上，先验证输入一致性 |

#### 7.5.5 可调参数

默认启用，仅需覆盖默认值时写：

- `min_match_len`（默认 128）：最小有效匹配长度
- `search_topk`（默认 3）：输出前 K 个最长匹配段
- `algorithm`（默认 `rolling_hash`）：可选 `suffix_array`

### 7.6 浮点位域比对（符号位 / 指数位 / 尾数位）

**原理：** 对浮点数据按位域拆开（IEEE 754 格式），分别比对**符号位（sign）**、**指数位（exponent）**、**尾数位（mantissa）**，定位差异发生在哪个位域。

**每个位域差异的含义：**

| 位域 | 若此位域差 | 指向 |
|---|---|---|
| 符号位 | 方向反了 | 符号处理 bug（减法 / 负数处理）；或极小值附近数值抖动 |
| 指数位 | 量级不同 | 溢出 / 下溢；量化尺度问题；FP16 表达范围超限 |
| 尾数位 | 精度不同 | 累加顺序差异；舍入模式差异；中间计算精度不足 |

**输出内容：**

- 三个位域各自的差异统计（差异元素数、最大 bit 差异、平均 bit 差异）
- 哪个位域差异最大（主导差异源）
- 可视化：三个位域差异分布对比图

**可调参数**（默认启用，dtype 按张量自动识别，仅需覆盖时写）：`report_topk`（默认 10）—— 差异最大的前 N 个元素详情。

### 7.7 QSNR 比对

**原理：** 把实测数据和 golden 都转为 **double（fp64）** 后计算 **QSNR（Quantization Signal-to-Noise Ratio，量化信噪比）**。

**公式（基本形式）：**

```
QSNR = 10 · log10( P_signal / P_noise )

其中：
  P_signal = Σ(golden²) / N
  P_noise  = Σ((golden - actual)²) / N
```

单位：dB。值越大表示差异越小。

**典型参考（仅供定位时参考，不作为通过标准）：**

| QSNR 范围 | 定位提示 |
|---|---|
| ∞ / 极高 | 差异极小，可能只是极少数元素低位差异 |
| 40+ dB | 差异量级不大，多为累积误差、舍入差异 |
| 20~40 dB | 中等差异，可能存在算子实现偏差 |
| < 20 dB | 严重偏差，通常是逻辑 bug 或数据格式错 |
| 负值 | 完全错乱，输入 / golden 对不上或模块挂死 |

**为什么转 double 计算：**

- 原始数据可能是 fp16 / bf16 / int8，直接算会因表达范围受限失真
- 转 double 后计算能准确反映差异的真实规模
- 与业界（TensorRT、CANN 等）比对工具的 QSNR 语义一致，便于横向对齐

**可调参数**（默认启用，统一转 double 计算，无需额外配置）。

### 7.8 辅助分析工具集成

精准比数裁决固定为 **bit 级一致**（唯一标准），无需配置。辅助分析的四种手段**默认全开**，仅在 bit 比对 FAIL 时触发；**无需在 yaml 里显式启用**。

只有在需要"关闭某个分析器"或"覆盖默认参数"时才写 yaml：

```yaml
# 大张量场景按需关闭 LCS 偏移比对（开销较大）；其他分析器继续生效
assistive_analysis:
  disable: [offset_compare]
  block_compare:
    block_size: 128             # 覆盖默认 64
```

默认参数：`block_size=64`、`offset_compare.min_match_len=128` / `search_topk=3`、`bit_field_compare` 按 dtype 自动选位域、`qsnr_compare` 自动转 double 计算。

### 7.9 综合报告结构

bit 比对 FAIL 时，辅助分析报告按统一结构输出：

```
精准比数报告 - <case_id>
├─ 裁决：FAIL（bit 级不一致）
│
├─ bit 比对摘要
│   ├─ 总元素数、差异元素数、差异率
│   └─ 差异元素首个位置
│
├─ 辅助分析结果
│   ├─ 分 block 比对：差异分布图 + 结论提示
│   ├─ 偏移比对（LCS）：最长匹配段 + 反推偏移量 + 次长匹配段
│   ├─ 位域比对：三位域差异占比 + 主导位域
│   └─ QSNR：数值 + 参考级别
│
└─ 综合定位建议
    根据四种分析结果的组合，给出"最可能的 bug 类别"提示
```

### 7.10 综合定位建议示例

工具根据辅助分析结果的组合给出**定位提示**，并非定论，仅辅助工程师判断：

| 分析组合 | 建议 |
|---|---|
| LCS 最长匹配段 ≈ 总长度 + 位域无差异 | 高置信度：**地址/布局错位 bug**，查 DMA 起址、padding、layout 转换 |
| LCS 出现多段较长匹配 + 各段 offset 不同 | **分段错位**，查多次 DMA 传输或多个内存块拼接逻辑 |
| 所有 block 均差 + QSNR 低 + 指数位主导差异 | **数值格式 / 量化问题**，查数据类型或尺度 |
| 某 block 集中差 + QSNR 高 + 尾数位主导差异 | **特定算子累积误差**，查该 block 对应算子实现 |
| 极少数元素差 + 符号位差异 | **边界/特殊值处理 bug**（0、NaN、极值附近）|
| QSNR 为负 / LCS 几乎无匹配 | **输入错位 / 模块挂死 / golden 不匹配**，先确认输入与 golden 一致性 |

### 7.11 使用边界

- **辅助分析不用于绕过 bit 比对**：即使 QSNR 极高、LCS 匹配度极高，bit 不一致仍判 FAIL
- **辅助分析有开销**：默认仅在 bit 比对 FAIL 时触发，避免拖慢常规回归
- **配置可裁剪**：大张量场景下 LCS 偏移比对开销较大，可按需关闭
- **结论仅为参考**：综合定位建议基于经验模式，最终 root cause 仍需工程师确认

### 7.12 多阶段场景：首发散阶段定位

阶段性比数会对多个阶段产出多份 dump。Autotest 自动对每份 dump 跑精准比数，定位第一个 FAIL 的阶段：

```
输入：有序阶段 dump 列表 S[0..n-1] 与对应 GOLDEN
输出：首发散阶段 index

for i in 0..n-1:
    verdict = precise_compare(S[i], GOLDEN[i])   # bit 级裁决（见 7.1）
    if verdict == FAIL:
        return i                                  # 找到首发散阶段
return -1                                          # 全部通过
```

输出包含：

- 阶段名（如 backbone_out / neck_out / head_out）
- 该阶段覆盖的层范围
- bit 比对摘要 + 辅助分析报告（见 7.9）
- 前一阶段的 PASS 数据（辅助判断上游污染）

### 7.13 比对工具入口

```bash
compare.py \
  --dump-dir dumps/cv_resnet50_cpu_baseline/ \
  --golden-dir golden/cv_resnet50_cpu_baseline/ \
  --mode stage_compare \                # stage_compare | end_to_end
  --output report/compare_001.html
# 辅助分析默认全开；大张量场景可加 --disable offset_compare
```

输出：

- HTML 报告（每阶段 bit 摘要 + 辅助分析图表）
- JSON 摘要（Autotest 读取用于裁决和自动切换决策）

### 7.14 端到端比数的简化路径

端到端模式只有一个比对项（最终输出 vs GOLDEN），无需"首发散定位"，但 bit 级判定 + FAIL 时辅助分析逻辑不变。

## 8. 自动化实现

### 8.1 Autotest 侧改造

**新增配置层：**

- 用例用 pytest markers 分类（`accuracy` / `perf` / `首用例` 等，见 00 · 4.4.6）
- 精度用例集 yaml 支持 `首用例` + `业务软件版本` 字段（见 3.5）
- 性能用例集 yaml 支持 `失败回落` block（见 3.5）
- dump 配置见 6.7

**新增执行编排：**

```
接收用例
    │
    ▼
读取策略（accuracy / perf / 首用例）
    │
    ▼
初始化模式
  - accuracy 首用例 → stage_compare
  - accuracy 正常规格 → end_to_end（首用例已过）
  - perf → end_to_end
    │
    ▼
下发 dump 配置到 CPU 桩
    │
    ▼
执行推理
    │
    ▼
收集 dump + 调比对工具
    │
    ▼
裁决
    │
    ▼
【自动切换决策】
  - accuracy 首用例通过 → 切 end_to_end
  - perf end_to_end 失败 & 自动回落启用 → 切 stage_compare 重跑
```

### 8.2 CPU 桩侧改造

- 新增 dump 控制寄存器（或命令接口），Autotest 可通过串口开关 dump
- 新增 dump 数据暂存区（独立 buffer）
- 增加批量文件传输能力（dump 数据回传主机）

### 8.3 平台软件 / 被测 FPGA 侧改造

- 在**阶段结束点**添加 hook（不是每层都加）
- hook 默认编译进二进制但运行期关闭，通过配置开关启用（零开销原则）
- 支持阶段列表动态注入

### 8.4 比对工具模块（compare 比数模块）

对应 **4.1.2 备选路径**的比数模块，**独立仓库管理**，跨平台 / 跨场景复用（link / rtl2c / fpga / emu / 调试分析都可调）：

```
compare/                    # 独立仓库，不与 Autotest 耦合
├── loaders.py              # 加载 npy + meta（支持 CPU 桩 dump 格式）
├── bit_compare.py          # bit 级判定（唯一裁决，见 7.1）
├── assistive/              # 辅助分析（仅在 FAIL 时触发，见 7.3~7.7）
│   ├── block_compare.py    # 分 block 比对
│   ├── offset_compare.py   # LCS 偏移比对
│   ├── bit_field.py        # 浮点位域比对
│   └── qsnr.py             # QSNR 比对
├── locator.py              # 首发散阶段定位（见 7.12）
├── switcher.py             # 模式切换决策（给 Autotest 调用）
├── visualizer.py           # 差异分布图、位域图
└── reporter.py             # 综合报告（HTML/JSON，见 7.9）
```

**独立仓库的意义：**

- 标准路径（接口 FPGA 硬件比对）裁决 FAIL 时，Autotest 自动切到备选路径取 dump 数据，调 compare 模块产出辅助分析报告
- 非 FPGA 平台（link / rtl2c / emu）天然走备选路径，直接复用 compare 模块
- 调试场景主机侧离线分析时独立使用，无需拉起 Autotest

## 9. 失败兜底

### 9.1 比数流程失败的手动步骤

| 失败场景 | 手动步骤 |
|---|---|
| dump 未产生 | 串口手动敲 `dump on --mode stage_compare`，执行短推理，从 CPU 桩取数据 |
| dump 数据不完整 | 检查 CPU 桩暂存区是否溢出，减少阶段数或粒度后重跑 |
| 比对工具执行失败 | 手动运行 `compare.py`，查堆栈 |
| golden 缺失 | 按基线对应的 GC 版本重跑 GC 硬件计算功能仿真出 GOLDEN（见子 Wiki 3 · 2.2）|
| 自动切换未生效 | 检查 Autotest 日志的切换决策；可人工下发下一用例 mode 配置 |
| 性能用例失败回落未触发 | 检查套件配置 `失败回落.mode` 字段；若为 manual，人工改脚本触发 |

### 9.2 手动阶段性比数最小步骤

```
1. 登录 CPU 桩串口
2. 确认平台软件已加载 DO 且 READY
3. 串口：dump config --mode stage_compare --stages backbone_out,neck_out,head_out
4. 串口：run_once --input <file>
5. 等 DONE
6. 串口：dump list                     # 查看产生的文件
7. 从 CPU 桩下载 dump 目录到主机
8. 主机运行 compare.py
```

### 9.3 手动切换模式

如果 Autotest 自动切换决策未按预期执行：

```bash
# 下一用例强制端到端
autotest run <next_case> --mode end_to_end

# 失败用例强制阶段性比数重跑
autotest rerun <failed_case> --mode stage_compare
```

## 10. 常见问题

| 现象 | 原因 | 处置 |
|---|---|---|
| 阶段性比数全部通过但端到端失败 | 后处理或输出解释层问题 | 检查模型外逻辑，看推理输出到最终业务结果之间的代码 |
| 首发散阶段定位到 backbone 末 | backbone 内部问题，需下钻 | 在该阶段内加细粒度 dump 点，再跑一次 |
| bit 比对 FAIL 但 QSNR 高 / LCS 匹配率高 | 少量低位差异，或整体错位但数值正确 | 不改变裁决（仍 FAIL）；按辅助报告定位（见 7.10），优先查舍入 / 累加顺序 / 地址布局 |
| bit 比对 FAIL 且 QSNR 为负 / LCS 几乎无匹配 | 输入 / golden 对不上，或模块挂死 | 先验证输入与 golden 一致性，再查模块状态 |
| 首用例通过但正常规格用例端到端失败 | 规格扩大后暴露的问题 | 对正常规格用例强制阶段性比数定位 |
| 性能用例端到端失败后自动回落没跑 | 套件 yaml 里 `失败回落.mode` 设成了 `off` 或 `manual` | 改回 `auto`（默认值），或手动触发 |
| dump 文件过大塞满 CPU 桩 | 阶段数过多或 shape 过大 | 减少阶段数、改粗粒度；首用例用 CPU 简化路径（天然无硬件 dump 负担）|
| golden 与当前输出都 0 但"通过" | 空数据误判（bit 级相等但语义无效） | 在 dump 元数据或输入准备层面加"非零占比"健壮性检查 |
| 自动切换到端到端后效率没提升 | 可能端到端 dump 配置仍开着某些开关 | 检查 Autotest 切换后的实际配置下发是否干净 |
| 辅助分析报告生成很慢 | 大张量的 LCS 偏移比对开销高 | 减小 `search_topk`、提高 `min_match_len`，或按需关闭 offset_compare |
