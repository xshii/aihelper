## 1. 术语与组件约定

### 1.1 核心组件定义

| 术语 | 一句话定义 |
|---|---|
| **SDK** | 外部发布的二进制库；所有使用平台都依赖 |
| **平台软件** | 平台侧核心运行环境（"运行时"），加载模型、接管业务执行 |
| **业务软件** | AI 业务逻辑实现，平台无关，部署到平台软件上运行 |
| **CPU 桩** | FPGA 链路嵌入式芯片侧的桩软件，触发接口 FPGA |
| **软调** | OS 接口插件，仅做调试性内存读写，**不支持函数执行** |
| **激励数据** | 用例输入（源头是主文档 4.0.2 出数产物），各平台按需转 binary 或 TXT |
| **LINK** | 功能仿真平台模型，**不支持 DEBUG** |
| **LINKST** | LINK 上的 CPU 桩特殊版本，**不支持动态交互**（共代码但无法拆步） |
| **VerFingerPrint** | 业务软件 DEBUG 日志的版本指纹字段，格式 `VerFingerPrint[<组件>]: <指纹>`，加载阶段校验 |
| **DO** | 业务软件 + 模型打包简称，加载到平台软件上运行 |
| **ESL** | 算子软件的性能仿真工具，EDA 版本依赖它出软件版本 |
| **算子软件** | EDA 平台的算子实现集合（EDA 专有） |
| **二阶段编译工具** | 第一次匹配 EDA TXT 格式；第二次匹配 EDA 业务执行 + hex（EDA 专有） |
| **CMC 制品仓库** | 上游分发 SDK / 平台软件 / 编译器，所有平台共享 |
| **smartci 仓库** | 跨业务组件融合仓：托管共享资源表 + 出包融合脚本 + LINK 冒烟（详见 1.5） |
| **业务组件 / 资源表** | 项目由 2 个业务组件组成，各自独立 CI；共享一张资源分配表（ID / 地址 / 约束）由 smartci 托管融合（详见 1.5） |

### 1.2 各平台版本组件对照速查

| 组件 | FPGA | LINK | EDA | RTL2C | EMU |
|---|:---:|:---:|:---:|:---:|:---:|
| SDK | ✅ 独立 | ✅ 内嵌平台软件 | — | — | 同 FPGA |
| 平台软件 | ✅ | ✅（内嵌 SDK） | — | — | 同 FPGA |
| 业务软件 | ✅ | ✅ | — | — | 同 FPGA |
| CPU 桩 | ✅ | ❌ 用 LINKST 替代 | — | — | 同 FPGA |
| LINKST | — | ✅ | — | — | — |
| 软调 | ✅ | ✅ | — | — | 同 FPGA |
| 激励数据 | ✅ 二进制，直接加载 | ✅ 二进制，通过 LINKST 加载 | ✅ TXT | ✅ TXT | ✅ 二进制，通过 TESGINE 加载 |
| ESL（性能仿真工具）| — | — | ✅（EDA 出版本依赖）| — | — |
| 算子软件 | — | — | ✅ | — | — |
| 二阶段编译工具 | — | — | ✅ | ✅（与 EDA 共享 pipeline，顺势产出 RTL2C 产物）| — |

> **上游通用组件：** SDK / 平台软件 / 编译器 由 **CMC 制品仓库**统一分发，所有平台共享，不由本项目构建。业务软件相关的打包脚本与跨模块共享资源由 **smartci 仓库**维护。
>
> EDA / RTL2C 列空白不代表没有 SDK / 业务软件，而是它们的版本制作以各自专有组件为主线；通用组件通过二阶段编译过程吸收进去，无需单独打包。

### 1.3 平台间依赖链

各平台版本不完全独立，存在以下依赖关系：

```
   CMC 制品仓库：SDK / 平台软件 / 编译器            （上游通用）
          +
   smartci 仓库：业务软件打包脚本 / 跨模块共享资源
                       │
        ┌──────────┬───┴─────────┬──────────────┬────────────┐
        ▼          ▼             ▼              ▼            ▼
     FPGA 版本   LINK 版本     EDA 版本      RTL2C 版本    EMU 版本
     (+ CPU 桩  (+ LINKST)    (算子软件 +    (EDA 二阶段  (暂同 FPGA,
     + 激励数据)              ESL 性能仿真 + 编译过程中    暂未启动)
                             二阶段编译)    顺势产出)
                                 │ │
                                 │ └─► RTL2C 与 EDA 共享构建上下文
                                 └──► 算子软件依赖 ESL 出版本
```

**关键点：**

- FPGA / LINK / EMU 共享上游（CMC / smartci）与平台无关核心软件；差异在**平台接入组件**（CPU 桩 ↔ LINKST）与加载方式
- **EDA 以算子软件 + ESL 性能仿真 + 二阶段编译为主线**，与 FPGA 组件集合差异大
- **RTL2C 与 EDA 共享 pipeline**，在 EDA 二阶段编译中顺势产出，而非先归档再转化

### 1.4 版本号规范

统一采用：`<baseline_id>-<date>-<commit_short>`，示例 `v1.2.3-20260421-a1b2c3d`。

各平台版本包都在 manifest 中记录相同的 `baseline_id` 与 `rtl_commit`，参见第 7 节。

### 1.5 smartci 仓的工作机制

本项目验证涉及 **2 个业务组件**（各自独立代码仓、独立 CI），但两者有一张**共享的资源表**需要共同维护、出包逻辑相同。**smartci 仓库**承担这部分"**跨业务组件的融合**"职责。

#### 1.5.1 smartci 仓托管的内容

| 内容 | 说明 |
|---|---|
| **共享资源表** | 两个组件共同使用的资源分配（ID / 地址 / 约束等）；由 smartci 托管为单一真相源 |
| **出包脚本** | 各业务组件二进制的打包脚本（保持两方的出包逻辑一致）|
| **融合脚本** | 把两个组件的二进制 + 融合后的资源表合并为一个完整 DO |
| **LINK 冒烟执行脚本** | 融合出的 DO 在 LINK 平台跑一遍冒烟，通过才算正式版本 |

#### 1.5.2 融合出包流程

smartci pipeline：从 CMC 拉双方制品 → 按合并规则融合资源表（CI 冲突检测：重复 ID / 地址冲突 / 约束不一致自动拦截）→ 执行融合出包脚本（两二进制 + 融合表 → 一个 DO）→ 本地 LINK 冒烟（必须通过）→ 产出正式 DO。冒烟失败直接阻断，DO 不下发。

> 资源表合并规则的演进（人工 review → 半自动 → 全自动）见 smartci 仓库内文档，本 wiki 不展开。

#### 1.5.3 为什么 LINK 冒烟是必经关卡

- **最快验证闭环**：LINK 分钟级，FPGA 小时级，冒烟放在 LINK 把"融合错了"挡在最早
- **保护下游资源**：融合失败的 DO 流到 FPGA / EMU 会浪费稀缺板级 / 机位资源
- **对齐主流程 4.9 流水线**：也是"link 作为准入闸门"这一业务设计的起点

#### 1.5.4 与各平台打包章节的关系

| 阶段 | 归属 |
|---|---|
| 业务组件各自二进制出包 + 上传 CMC | 业务组件自研仓（本 wiki 不覆盖，各组件 CI 自治）|
| 融合出包 + 资源表合并 + LINK 冒烟 | smartci 仓（本小节）|
| 融合后的 DO 加载到各平台 | 本 wiki 第 2 / 3 / 6 章（FPGA / LINK / EMU 加载流程）|

## 2. FPGA 平台版本制作与加载

FPGA 是链路最完整的平台，版本制作与加载流程也最详细。本章节作为后续其他平台的参照基准。

### 2.1 版本组件

FPGA 平台版本包含：

1. **SDK**（二进制发布物，**CMC 分发**；业务验证阶段**不必须感知**，可由业务软件裁剪后内嵌运行 —— 即不带 OS 直接跑业务软件的形态也成立）
2. **平台软件**（运行时核心，**CMC 分发**）
3. **业务软件**（AI 业务逻辑，本项目自研 + **smartci 仓库**维护打包脚本）
4. **CPU 桩**（嵌入式芯片软件，触发接口 FPGA）
5. **软调**（插件，内存读写）
6. **激励数据**（用例输入，FPGA 上为二进制，直接加载）

### 2.2 制作流程总览

```
  源码 / 外部 SDK
       │
       ▼
  ┌─────────────┐
  │ 2.3 SDK 入库 │
  └──────┬──────┘
         ▼
  ┌──────────────────┐
  │ 2.4 平台软件编译  │
  └──────┬───────────┘
         ▼
  ┌──────────────────┐
  │ 2.5 业务软件编译  │
  └──────┬───────────┘
         ▼
  ┌──────────────────┐
  │ 2.6 CPU 桩编译    │
  └──────┬───────────┘
         ▼
  ┌──────────────────┐
  │ 2.7 软调插件构建  │
  └──────┬───────────┘
         ▼
  ┌───────────────────────┐
  │ 2.8 激励数据准备      │
  │ （对接公共前置"出数"）│
  └──────┬────────────────┘
         ▼
  ┌──────────────────┐
  │ 2.9 联合打包      │
  └──────┬───────────┘
         ▼
  ┌──────────────────┐
  │ 2.10 产物校验     │
  └──────┬───────────┘
         ▼
     入制品库
```

### 2.3 SDK 获取（CMC 分发，不构建）

- 渠道：从 **CMC 制品仓库**按版本号拉取（非本项目构建）
- 完整性校验：MD5 + 签名
- 本地按 `sdk/<version>/` 归档缓存
- 自动化：`fetch_from_cmc.sh --component sdk --version X.Y.Z`
- 手动兜底：从 CMC 页面下载，人工校验
- **注意**：业务验证阶段不一定上传 SDK，业务软件可裁剪后内嵌；上传与否由构建配置决定

### 2.4 平台软件获取（CMC 分发，不构建）

- 渠道：从 **CMC 制品仓库**按版本号拉取（非本项目构建）
- 产物：`platform_sw_vX.Y.Z.tar.gz`
- 自动化入口：`fetch_from_cmc.sh --component platform_sw --version X.Y.Z`

### 2.5 业务软件编译

- 源码同步
- 依赖 SDK + 平台软件头文件（从 CMC 拉取的产物）
- **打包脚本与跨模块共享资源来自 smartci 仓库**
- 产物：`biz_sw_vX.Y.Z.tar.gz`
- 自动化入口：`build_biz.sh TARGET=fpga`
- **可选**：裁剪模式下将 SDK 运行时必需部分内嵌，产出可不带 OS 直接运行的业务软件包

### 2.6 CPU 桩编译

- 源码同步（CPU 桩在独立 repo 或子目录）
- 编译到嵌入式芯片目标架构
- 功能：触发接口 FPGA、与 RDO/Autotest/串口的交互
- 产物：`cpustub_vX.Y.Z.bin`
- 自动化入口：`build_cpustub.sh TARGET=fpga`

### 2.7 软调插件构建

- 源码同步（软调以插件形式维护）
- 功能限制说明：**仅支持内存读写，不支持函数执行**
- 产物：`softdbg_plugin_vX.Y.Z.so` 或等效格式
- 自动化入口：`build_softdbg.sh`

### 2.8 激励数据准备

- 激励数据来源于公共前置的"出数"环节（见主文档 4.0.2）
- 两种形式：
  - **实时出数**：CI 触发时动态生成
  - **静态出数**：引用已版本化的数据集
- **FPGA 消费格式：二进制**（EDA / RTL2C 走 TXT，EMU 走二进制但通过 TESGINE 灌入；源头同一份，各平台按需转换）
- 入 FPGA 版本包时打包为 `stim_data_<case_set>.dat`

### 2.9 联合打包

将以上全部产物合并为 FPGA 版本包：

```
fpga_package_vX.Y.Z/
├── sdk/                         # 来自 2.3
├── platform_sw/                 # 来自 2.4
├── biz_sw/                      # 来自 2.5
├── cpustub/
│   └── cpustub.bin              # 来自 2.6（二进制固件）
├── softdbg/
│   └── softdbg_plugin.so        # 来自 2.7
├── stim_data/                   # 来自 2.8
│   └── *.dat
├── VERSION
└── manifest.yaml
```

`manifest.yaml` 记录 `baseline_id`、`rtl_commit`、各组件版本号与哈希。

### 2.10 产物校验

- 版本号一致性（manifest vs 各组件 VERSION 文件）
- 哈希校验
- 冒烟自检（可选）：在 CI 机器上拉起 FPGA 模拟器或干跑 CPU 桩自检命令
- ABI 兼容性（SDK 与平台软件）
- **版本指纹（VerFingerPrint）打点检查**：每个组件在启动 DEBUG 日志中打印版本指纹，格式 `VerFingerPrint[<组件>]: <指纹字符串>`（例如 `VerFingerPrint[biz_sw]: 65010219930822401X`），Autotest 在版本加载阶段解析该日志验证组件版本匹配（见主文档 4.3）

### 2.11 版本加载到 FPGA 平台

入口：`autotest load --baseline <baseline_id> --target fpga --device <id>`，底层按下表逐步执行：

| 步骤 | 命令（示例） | 预期日志关键字 | 失败定位 |
|---|---|---|---|
| 0. 硬复位 FPGA（仅硬切换） | `reset_fpga --hard` | — | — |
| 1. 上传 SDK | `tar -xzf sdk.tar.gz` | `SDK init done` | 版本不匹配 |
| 2. 启动平台软件 | `platform_main --config xxx.yaml &` + `wait_ready --timeout 60` | `platform ready` | 依赖 / 端口占用 |
| 3. 上传业务软件 | `load_biz_sw --file biz_sw.tar.gz` | `biz loaded` | 接口版本不匹配 |
| 4. CPU 桩固件（按需） | `flash_cpustub --file cpustub.bin` | `cpustub alive` | 嵌入式芯片状态异常 |
| 5. 软调插件 | `load_softdbg --file softdbg_plugin.so` | `softdbg attached` | OS 接口变化 |
| 6. 激励数据 | `load_stim --file stim_data.dat` | `stim ready` | 文件损坏 / 路径错 |
| 7. 状态校验 | `check_status` | `READY` | — |

> **失败兜底统一动作**：保存日志 + 寄存器快照 → 停止已启动进程 → 硬复位 FPGA → 按上表"失败定位"列回查。具体命令以实际为准。

## 3. LINK（功能仿真）平台版本制作与加载

### 3.1 版本组件

LINK 平台版本包含（SDK / 平台软件 / 编译器的上游来源与 FPGA 完全一致，均来自 CMC）：

1. **平台软件**（CMC 分发；SDK 按构建配置内嵌或独立）
2. **业务软件**（smartci 仓库提供打包脚本）
3. **LINKST**（替代 CPU 桩，因为 LINK 不支持 DEBUG）
4. **软调**
5. **激励数据**（二进制，**通过 LINKST 加载**，不是直接加载）

### 3.2 制作流程总览

源码 → 平台软件编译（内嵌 SDK）→ 业务软件编译 → **LINKST 构建**（§ 3.4，与 FPGA CPU 桩共代码不同形态）→ 软调插件构建 → 激励数据准备 → 联合打包入制品库。各通用步骤参照 FPGA § 2.3-2.8，差异点见 § 3.3。

### 3.3 LINK 与 FPGA 的差异

平台软件获取 / 业务软件编译 / 软调插件构建 / 激励数据**全部与 FPGA 共代码**，只是编译目标 / 产物名不同（详见 § 2.3-2.4 / § 2.5 / § 2.7 / § 2.8 对应部分）。下面只列差异：

| 差异点 | LINK 处理 | FPGA 处理 |
|---|---|---|
| SDK 是否内嵌平台软件 | 默认内嵌（简化加载） | 不内嵌 |
| 替代 CPU 桩 | **LINKST**（§ 3.4） | CPU 桩 |
| 激励数据加载 | LINKST 预加载（启动前） | 运行时注入 |
| 产物名前缀 | `*_link_vX.Y.Z.*` | `*_vX.Y.Z.*` |
| 自动化入口 | `--target link` / `TARGET=linkst` | 默认或 `TARGET=cpustub` |

### 3.4 LINKST 构建

- **共 CPU 桩代码，但编译为 LINKST 形态**：因为 LINK 不支持 DEBUG，LINKST 承担 CPU 桩在 FPGA 平台上的用例启动等职能
- **重要限制 — LINKST 不支持动态交互**：用例启动所需动作必须预置为静态序列；调试方式受限，只能事后看日志
- 产物：`linkst_vX.Y.Z.bin`；自动化入口：`build_cpustub.sh TARGET=linkst`

### 3.5 联合打包与加载

打包结构与 FPGA 平行：`link_package_vX.Y.Z/` 包含 `platform_sw/`（内嵌 SDK）/ `biz_sw/` / `linkst/linkst.bin` / `softdbg/softdbg_plugin.so` / `stim_data/*.dat` / `VERSION` / `manifest.yaml`。

加载顺序：启动 LINK 仿真 → 加载平台软件（含 SDK）→ 加载业务软件 → 挂 LINKST → 用 LINKST 预加载激励数据 → 加载软调插件 → 用例执行。

> **失败兜底受 LINKST 限制**：不支持动态交互 → Autotest 只能一次性下发全部配置；失败只能事后看日志，不能像 FPGA 那样中途查状态。兜底动作仅限日志离线分析、重编 LINKST 配置、重启 LINK。

## 4. EDA 平台版本制作

### 4.1 版本组件

EDA 平台版本与 FPGA / LINK 的组件集合差异很大，包含：

1. **算子软件**（依赖 **ESL** 进行性能仿真出版本；ESL 本身不是 EDA 专有，是算子软件的性能仿真工具）
2. **二阶段编译工具**（EDA 与 RTL2C 共享；RTL2C 产物在此 pipeline 中顺势产出，具体流程见 4.6）
3. **激励数据（TXT 格式）**：由出数产物转换而来，EDA 消费的是 TXT 文本

> EDA 平台的 SDK / 业务软件等通用组件通过二阶段编译过程消化进去，不独立打包。

### 4.2 EDA 的定位回顾

EDA **不是执行平台**，Autotest 不在 EDA 中跑用例。EDA 版本是**交付物**，供相关人员在 EDA 环境中手动进行波形分析。因此本章节只描述**版本制作**，不描述"加载"。

### 4.3 制作流程

```
  源码 + CMC 组件
       │
       ▼
  ┌───────────────────────┐
  │ 4.4 ESL 性能仿真      │
  └──────┬────────────────┘
         ▼
  ┌───────────────────────┐
  │ 4.5 算子软件构建（依赖 ESL 出版本）│
  └──────┬────────────────┘
         ▼
  ┌──────────────────────────────────────────────┐
  │ 4.6 二阶段编译                                │
  │   ├─ 第一次：ESL 流程 + 自定义数据导出（TXT）│
  │   └─ 第二次：业务执行 + 二进制 hex             │
  └──────┬───────────────────────────────────────┘
         ▼
  ┌──────────────────┐
  │ 4.7 打包与上传    │
  └──────────────────┘
```

### 4.4 ESL 性能仿真

- ESL 是算子软件的性能仿真工具，**非 EDA 专有**
- 按指定 `rtl_commit` 运行性能仿真，产出算子级性能数据
- 产物：ESL 仿真报告 + 后续算子软件构建所需的输入

### 4.5 算子软件构建

- EDA 平台的算子实现集合，**依赖 ESL 性能仿真结果出版本**
- 构建并验证可用
- 产物：算子软件包

### 4.6 二阶段编译

**第一次编译（匹配 EDA 数据格式）**：
- **标准 ESL 流程 + 自定义数据导出脚本**
- 目的：产出 EDA 所需的数据格式（**TXT 文本**）
- 产物：ESL 标准产物 + EDA 数据 TXT

**第二次编译（匹配 EDA 业务执行流程 + hex）**：
- 匹配 EDA 业务执行流程与二进制 hex 流程
- 可以跑启动流程，导出**启动参数 TXT** 和**二进制 hex**
- **限制**：由于 memmap 等差异，第二次编译产物**无法运行完整业务流程**，仅用于 EDA 分析 / RTL2C 顺势产出

### 4.7 打包与上传

```
eda_package_vX.Y.Z/
├── esl/                          # 4.4 ESL 版本
├── operators/                    # 4.5 算子软件
├── build_stage1/                 # 4.6 第一次编译产物（含数据 TXT）
├── build_stage2/                 # 4.6 第二次编译产物（启动参数 TXT + 二进制 hex）
├── stim_data/                    # 激励数据 TXT（由出数产物转换，EDA / RTL2C 共用）
│   └── *.txt
├── VERSION
└── manifest.yaml                 # 含 rtl_commit
```

打包压缩后，Autotest 自动上传至指定位置，供人工分析。

### 4.8 触发时机

Autotest 对 EDA 版本的打包触发有两种方式：

- **自动**：主回归（fpga / rtl2c）失败时自动触发打包上传
- **人工**：相关人员按需手动触发

## 5. RTL2C 平台版本制作

### 5.1 版本组件

RTL2C 平台**没有独立的构建组件**，产物在 EDA 二阶段编译过程中**顺势生成**。其消费的**激励数据为 TXT 格式**（与 EDA 一致，来自同一份出数产物的文本转换）。

### 5.2 生成方式

**RTL2C 不依赖 EDA 归档版本做二次转化**，而是**在 EDA 构建 pipeline 中直接产出**：

- RTL2C 与 EDA 共享同一次编译的上下文（同一 `baseline_id` / `rtl_commit`）
- EDA 二阶段编译运行过程中，同步产出 RTL2C 所需格式
- **无需独立的"EDA 二次转化工具"步骤**

### 5.3 制作流程

```
   EDA 构建 pipeline (见第 4 章)
      │  （二阶段编译运行过程中）
      ▼
  ┌────────────────────────────────┐
  │ 5.4 在 EDA 构建中顺势产出       │
  │     RTL2C 所需格式              │
  └──────┬─────────────────────────┘
         ▼
  ┌──────────────────┐
  │ 5.6 产物校验      │
  └──────┬───────────┘
         ▼
  ┌──────────────────┐
  │ 5.7 打包入库      │
  └──────────────────┘
```

### 5.4 在 EDA 构建中顺势产出

- RTL2C 与 EDA 使用同一 `baseline_id` / `rtl_commit`
- EDA 二阶段编译运行过程中同步产出 RTL2C 可执行的格式
- 无需从 EDA 归档版本二次拉取，无需独立的转化工具

### 5.6 产物校验

- 转化后产物的完整性
- RTL2C 脚本在沙箱环境一键自测通过

### 5.7 打包入库

```
rtl2c_package_vX.Y.Z/
├── converted/                    # 5.5 转化后产物
├── scripts/                      # RTL2C 执行脚本
├── VERSION
└── manifest.yaml                 # 含 rtl_commit + 依赖的 eda_package 版本
```

### 5.8 版本加载

RTL2C 没有复杂的"加载"动作，Autotest 执行时直接拉取 rtl2c_package，按脚本执行。平台业务职责见 [子 Wiki 5 · 第 2 章](./05_子wiki_多执行平台架构扩展.md)；回归中的位置见主文档 4.8 / 4.9。

## 6. EMU 平台版本制作与加载

> **备注：EMU 平台暂未启动**。本章作为后续启动时的参照，当前回归不依赖 EMU，相关章节内容为规划性描述。
>
> 另：**EMU 支持 checkpoint / replay**（长推理中间态保存，断点续跑），便于调试；FPGA 是否支持此能力**待确认**。

### 6.1 版本组件

EMU 平台版本**暂同 FPGA**，组件集合完全相同：

1. SDK
2. 平台软件
3. 业务软件
4. CPU 桩
5. 软调
6. 激励数据（二进制，通过 TESGINE 灌入）

### 6.2 与 FPGA 的差异

虽然**版本制作内容相同**，但**加载链路不同**（详见主文档 4.8.2 节）：

| 动作 | FPGA | EMU |
|---|---|---|
| 命令通道 | RDO → CPU 桩 | EMU 专用 API → espi → CPU 桩 |
| 数据通道 | CPU 桩 → 接口 FPGA | TESGINE 直接灌入 |
| 是否经过接口 FPGA | ✅ | ❌ |

### 6.3 制作流程

完全参照第 2 章 FPGA 制作流程（§ 2.3-2.10），仅产物包名改为 `emu_package_vX.Y.Z/`。

### 6.4 版本加载到 EMU 平台

#### 6.4.1 加载顺序

```
1. 启动 EMU 硬件仿真器
2. 通过 EMU 专用 API 建立连接
3. 通过 espi 将 CPU 桩固件下发到嵌入式芯片
4. 上传 SDK、平台软件、业务软件
5. 上传软调插件
6. 通过 TESGINE 灌入激励数据（不经过接口 FPGA）
7. 状态校验（READY）
```

#### 6.4.2 通道差异与失败兜底

EMU 加载顺序逻辑与 FPGA 相似，但每一步底层通道不同：FPGA 走 RDO + CPU 桩文件传输，EMU 走 EMU API + espi 命令 + TESGINE 数据注入。Autotest 的 emu_adapter 与 fpga_adapter 因此独立实现（见 [子 Wiki 5 · 2.4](./05_子wiki_多执行平台架构扩展.md)）。

兜底排查参照 FPGA，但工具链不同：EMU API 有自己的诊断命令，espi 异常和 TESGINE 灌数据失败是 EMU 特有失败类型。

### 6.5 后续演进

"暂同 FPGA"意味着 EMU 版本制作当前没有特殊化，但随着 EMU 平台特化需求出现（例如性能测量专用 profile、不同的硬件 trace 机制），未来可能会引入 EMU 独有组件。届时本章节需要扩展。

## 7. 版本基线与跨平台关系

### 7.1 统一基线

一次完整的回归执行对应一个**版本基线**，基线 manifest 同时引用 5 个平台的版本包：

```yaml
baseline_id: v1.2.3-20260421-a1b2c3d
rtl_commit: abc1234                  # 所有平台共享

platform_packages:
  fpga:
    package: fpga_package_v1.2.3
    components:
      sdk: sdk_v1.4.0
      platform_sw: platform_sw_v1.2.3
      biz_sw: biz_sw_v1.2.3
      cpustub: cpustub_v1.2.3
      softdbg: softdbg_v1.2.3

  link:
    package: link_package_v1.2.3
    components:
      platform_sw: platform_sw_link_v1.2.3    # CMC 分发，构建配置可选内嵌 SDK
      biz_sw: biz_sw_v1.2.3
      linkst: linkst_v1.2.3
      softdbg: softdbg_v1.2.3

  eda:
    package: eda_package_v1.2.3
    components:
      operators: operators_v1.2.3             # 依赖 ESL 出版本
      two_stage_compiler: compiler_v1.2.3

  rtl2c:
    package: rtl2c_package_v1.2.3
    shared_with_eda: true                     # 在 EDA 二阶段编译中顺势产出
    # 无独立组件

  emu:
    package: emu_package_v1.2.3
    components:                               # 暂同 FPGA
      sdk: sdk_v1.4.0
      platform_sw: platform_sw_v1.2.3
      biz_sw: biz_sw_v1.2.3
      cpustub: cpustub_v1.2.3
      softdbg: softdbg_v1.2.3
```

### 7.2 同一 `rtl_commit` 约束

所有平台版本包必须基于同一个 `rtl_commit` 构建。这是跨平台精确比对（bit-exact）的前提，详见 [子 Wiki 5](./05_子wiki_多执行平台架构扩展.md) 与 [子 Wiki 6](./06_子wiki_Autotest软件架构.md)。

CI 强制检查：基线 manifest 生成前，校验各平台包的 `rtl_commit` 字段完全一致，否则拒绝合成基线。

### 7.3 构建 pipeline 编排

```
RTL commit
   │
   ├─► pipeline-fpga       (综合+布局布线，小时级)
   ├─► pipeline-emu        (emu 工具链，分钟到十分钟级；暂未启动)
   ├─► pipeline-link       (编译+LINKST 构建，分钟级)
   └─► pipeline-eda+rtl2c  (ESL 仿真 + 二阶段编译，小时级；EDA 与 RTL2C 共享构建上下文，顺势同时产出)
```

**RTL2C 在 EDA pipeline 中顺势产出**，不是先等 EDA 归档再二次转化，因此不存在"必须在 EDA 完成后才能启动"的串行约束 —— 两者是同一 pipeline 的两个输出。

### 7.4 版本制作失败对回归编排的影响

| 平台构建失败 | 对回归的影响 |
|---|---|
| fpga 构建失败 | 阻断 fpga 主回归 |
| emu 构建失败 | 阻断性能出口；诊断分支受影响（EMU 暂未启动，当前影响有限）|
| link 构建失败 | 阻断准入闸门，整轮回归无法开始 |
| eda 构建失败 | RTL2C 连带失败（共享 pipeline）；诊断分支的 eda 产出不可用 |
| rtl2c 构建失败 | link ‖ rtl2c 并行路径之一失败，波形定位受影响 |

## 8. 常见问题

| 现象 | 可能原因 | 处置 |
|---|---|---|
| FPGA 加载 SDK 成功但平台软件启动失败 | CMC 拉取的 SDK 与平台软件版本错配 | 检查 manifest 的版本绑定；重新从 CMC 拉取 |
| LINK 平台找不到激励数据 | LINKST 没加载成功 | 检查 LINKST 构建日志；因不支持动态交互，通常需要重来一次 |
| LINK 平台调试困难 | LINK 不支持 DEBUG | 依赖离线日志；必要时切 rtl2c 或 emu 平台复现 |
| RTL2C 产物不完整 | EDA 二阶段编译未完成或出错（RTL2C 与 EDA 共享 pipeline）| 先修 EDA 二阶段编译，RTL2C 顺势产出即可恢复 |
| VerFingerPrint 打点与期望不一致 | 组件版本错配或日志解析异常 | 对齐 manifest 指定的各组件版本；检查日志格式 `VerFingerPrint[<组件>]: ...` |
| EMU 加载成功但 TESGINE 灌数失败 | 数据格式或通道未准备好 | 查 EMU API 日志；确认 TESGINE profile 与本版本匹配 |
| EDA 包上传后仍无法用于分析 | 打包时遗漏依赖（例如算子软件版本不符，或 ESL 仿真数据缺失）| 检查 manifest 完整性；重新打包 |
| 基线合成被拒绝 | 各平台包 `rtl_commit` 不一致 | 回到构建 pipeline 检查，确保所有平台基于同一 commit |
| CPU 桩加载成功但无响应 | 嵌入式芯片复位状态异常 | 硬复位 FPGA；查 RDO 连接 |
| 软调挂载失败 | OS 接口变化或权限不足 | 查 OS 版本；确认软调插件兼容性 |
