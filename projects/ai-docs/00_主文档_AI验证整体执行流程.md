## 第 1 章　文档说明

### 1.1 目的与范围

本文档规范 AI 验证软件在 FPGA 原型环境下的自动化执行流程，覆盖版本制作、部署、用例执行、中间结果分析、结果判定、异常定位全链路，作为团队内部执行准则与交接依据。

### 1.2 读者

- 验证开发人员（编写/维护自动化用例）
- 验证执行人员（日常回归与问题定位）
- 芯片/软件开发人员（读懂验证结果、配合定位）

### 1.3 术语与缩写

| 术语 | 含义 |
|---|---|
| Autotest | 用例执行自动化平台；主体基于 **pytest** 框架（见 4.4.4） |
| RDO | IP 远程盒子，连接 FPGA 原型环境的近端 DEBUG 端口 |
| CPU 桩 | 硬件模块，代表主控 CPU，与接口 FPGA 交互；其上的流程由 Autotest 编排 |
| 接口 FPGA | 承载对外接口逻辑的 FPGA；**标准比数路径**的硬件比对实现方 |
| 被测 FPGA | 承载被测 AI 芯片逻辑的 FPGA |
| 平台软件 | 平台侧的核心运行环境，也称"运行时"；CMC 制品仓库分发 |
| 业务软件 | AI 业务逻辑的实现，本项目自研；打包脚本与跨模块共享资源来自 smartci 仓库；精度阶段性比数需求下有三个配合版本（初级 / 进阶 / 商用，见子 Wiki 2 · 第 4 章）|
| DO | 业务软件 + 模型打包后的简称，加载到平台软件上运行 |
| 软调 | OS 提供的底层内存读写接口插件；**仅支持读写特定内存地址，不支持函数执行**；**备选比数路径**依赖软调 |
| CMC 制品仓库 | 上游制品库，分发 SDK / 平台软件 / 编译器（不由本项目构建）|
| smartci 仓库 | 维护业务软件打包脚本与跨模块共享资源 |
| VerFingerPrint | 业务软件 DEBUG 日志中的版本指纹字段，格式 `VerFingerPrint[<组件>]: <指纹>`（见 4.3）|
| 精准比数 | 输出张量与 golden 的 **bit 级一致比对**（唯一通过标准）；FAIL 时附带多种辅助分析（分 block / LCS 偏移 / 位域 / QSNR）提升定位效率，辅助结果不改变裁决。见 [子 Wiki 2 · 第 7 章](./02_子wiki_中间结果软调.md) |
| 阶段性比数 | 业务软件按预先规划的 ID 导出部分中间张量比对 GOLDEN（精度定位 / 首用例准入用）|
| 端到端比数 | 使用原始业务软件，不做任何中间 dump，仅比对最终输出（性能用例与常规回归默认）|
| 标准比数路径 | 被测 FPGA 与接口 FPGA 硬件配合完成 bit 精准比对；CPU 桩 ↔ 被测 FPGA 交互走接口 FPGA 消息通道（见子 Wiki 2 · 4.1）|
| 备选比数路径 | CPU 桩通过软调直读被测 FPGA 内存做 dump，由主机侧 **compare 比数模块**（独立仓库）比对；消息通道不通时使用 |
| 首用例 | 精度用例集的打头阵用例，**CPU 简化实现 + 关闭硬件优化特性**（见子 Wiki 2 · 5.1）|
| 原型版本 | 当前回归使用的 FPGA 原型板版本标识，5.2 命名规范的首字段 |
| Golden 数据 | 用于比对的参考输出；来自 **GC 硬件计算功能仿真**（非 CPU/GPU 参考），以**硬件目标格式**存储（见子 Wiki 3 · 2.2）|
| DFX | Design for X，本文档特指硬件可测试性 / 告警监控能力 |

## 第 2 章　验证架构总览

### 2.1 硬件拓扑

```
┌─────────────┐      telnet / RDO 串口     ┌───────────┐
│  Autotest   │ ◄─────────────────────────► │   RDO     │
│   平台      │                             │ (IP 远程)  │
└─────────────┘                             └─────┬─────┘
                                                  │ 近端 DEBUG
                                                  ▼
                                             ┌─────────┐
                                             │ CPU 桩  │
                                             └────┬────┘
                                                  │
                                                  ▼
                                             ┌──────────┐
                                             │ 接口 FPGA │
                                             └────┬─────┘
                                                  │
                                                  ▼
                                             ┌──────────┐
                                             │ 被测 FPGA │
                                             │ (AI 逻辑) │
                                             └──────────┘
```

**数据流向：**

- **下行**：Autotest → RDO → CPU 桩 → 接口 FPGA → 被测 FPGA（命令、配置、数据）
- **上行**：被测 FPGA → 接口 FPGA → CPU 桩 → 串口回显/文件下载 → Autotest（状态、日志、结果、dump 数据）

> 上图为 **FPGA 原型（上板）** 平台的硬件拓扑。Autotest 共支持 5 个平台：4 个可执行平台（link / rtl2c / fpga / emu）+ 1 个版本交付平台（eda，只打包上传供人工分析，不在 Autotest 内执行）。平台概览见 4.0 节；平台职责与业务需求见 [子 Wiki 5](./05_子wiki_多执行平台架构扩展.md)；各平台版本加载细节见 [子 Wiki 1](./01_子wiki_版本制作与加载.md)；软件架构与 adapter 实现见 [子 Wiki 6](./06_子wiki_Autotest软件架构.md)。

**注意 EMU 链路与 FPGA 不同**：EMU 使用自有 API 接入，通过 **espi** 控制 CPU 桩，通过 **TESGINE** 等方式灌数据，**没有接口 FPGA**。

### 2.2 Autotest 软件架构（简述）

> 本节只交代架构现状与扩展方向；完整的分层设计、端口定义、扩展点规范、演进路线见 [子 Wiki 6：Autotest 软件架构](./06_子wiki_Autotest软件架构.md)。本主文档聚焦**业务验证全执行流程**与 **Autotest 实操**，架构细节不在此展开。

#### 2.2.1 现状与扩展需求

- **现状**：当前 Autotest **只支持 FPGA 原型（上板）调试**，其它平台（link / rtl2c / emu / eda）的联动尚未打通
- **扩展需求**：
  - **EMU 是必须扩展的** —— FPGA 测性能不准，EMU 才是性能用例权威出口；FPGA 失败的诊断也依赖 EMU。**EMU 自有 API 提供 Python 接口**，可直接 import 集成进 pytest 主体，不需要跨语言桥接
  - **其他平台（link / rtl2c / eda）也需要扩展**才能参与联动（link 准入、rtl2c 波形辅助定位、eda 打包交付）
  - 平台扩展不是"加几个脚本"可以解决 —— 必须做**软件架构优化**，让新平台接入成本可控

#### 2.2.2 架构优化方向：领域分层

按"核心领域 → 能力服务 → 平台适配"三层组织（依赖方向单向，扩展点永远在最外层 adapter），新增平台不动核心。详细分层、端口定义、PAL 关系、演进路线见 **[子 Wiki 6](./06_子wiki_Autotest软件架构.md)**。


### 2.3 组件构成与加载关系

本项目验证涉及以下组件（无容器镜像概念）：

| 组件 | 形态 | 来源 | 加载方式 |
|---|---|---|---|
| SDK | 二进制发布物 | **CMC 制品仓库**分发（不由本项目构建）| 可选上传到 CPU 桩；业务软件也可裁剪后内嵌 SDK |
| 平台软件（运行时）| 核心运行环境 | **CMC 制品仓库**分发 | 上传到 CPU 桩 |
| 编译器 | 模型转换工具链 | **CMC 制品仓库**分发 | 主机侧运行，产出目标格式模型 |
| 业务软件 | AI 业务逻辑 | 本项目自研 + **smartci 仓库**（维护打包脚本与跨模块共享资源）| 上传 → 加载 → 启动推理 |
| CPU 桩 / 软调 | 嵌入式芯片软件 / 内存读写插件 | 本项目自研 | 按需上传 |

组件版本绑定与打包流程详见 **子 Wiki 1**。

### 2.4 AI 验证的特殊性

相对于通用硬件验证，AI 验证存在如下差异，构成 Autotest 扩展的必要性：

- **输出必须 bit 级完全一致**（精准比数的唯一通过标准，不允许任何容忍度绕过）；FAIL 时启用辅助分析提升定位效率，但辅助结果不改变裁决
- **中间结果有物理意义**（每层张量对应神经网络特定语义），需支持阶段性比数
- **性能与精度不可同时观察**——阶段性比数会干扰计时，所以性能用例默认端到端比数，失败时才回落阶段性
- **Golden 来自 GC 硬件计算功能仿真**（不用 CPU / GPU 参考——只有 GC 能产出 bit 级与硬件一致的输出）；存储格式为硬件目标格式；每基线需匹配对应 GC 版本（见 [子 Wiki 3 · 2.2](./03_子wiki_业务结果确认.md)）

扩展能力概述见第 6 章，细节见 **子 Wiki 2**。

## 第 4 章　Autotest 主流程

本章描述 Autotest 的执行流程。Autotest 支持多个执行平台，各平台有共同的前置步骤，但执行步骤不同。本章先讲公共前置与平台概览（4.0），再详细展开 FPGA 平台的完整流程（4.1~4.7），其他平台流程对照表见 4.8，整体回归编排见 4.9。其他平台的版本加载流程见 [子 Wiki 1](./01_子wiki_版本制作与加载.md)，平台业务职责与需求见 [子 Wiki 5](./05_子wiki_多执行平台架构扩展.md)。

### 4.0 执行平台概览

#### 4.0.1 5 个执行平台

Autotest 识别 **5 个平台**，分为 **4 个可执行平台 + 1 个版本交付平台**：

| 平台 | 角色 | Autotest 内执行？ | 主要用途 |
|---|---|---|---|
| **link**（功能仿真） | 可执行 | ✅ | 准入、L0 冒烟、快速功能验证 |
| **rtl2c** | 可执行 | ✅ | RTL 等价性、主回归的一路 |
| **fpga**（上板） | 可执行 | ✅ | 主回归的另一路、功能验证主力 |
| **emu**（硬件仿真器） | 可执行 | ✅ | FPGA 失败诊断 + **性能出口**（FPGA 性能不准） |
| **eda** | 交付 | ❌（仅打包上传，人工分析） | 失败场景波形分析 |

#### 4.0.2 公共前置步骤

所有可执行平台在进入各自流程前，统一完成以下两步：

**① 用例选择**

- 按用例级别（L0/L1/L2/L3）、模型类型、标签筛选
- 决定目标平台（默认按 4.0.1 与用例属性自动匹配）
- 输出：本次执行的用例清单

**② 出数（输入 + golden 准备）**

"出数" = 准备本次用例所需的输入数据以及对应的 golden 参考输出，支持两种形式：

| 形式 | 说明 | 适用 |
|---|---|---|
| **实时出数** | 运行时动态调用出数工具生成数据 | 新增用例、随机/参数化输入 |
| **静态出数** | 引用已预生成并版本化的数据集 | 回归用例、确定输入 |

配置示例：

静态出数（默认，二进制小工具按版本号下载，版本号含软件分类）：

```yaml
data_source:
  fetch_tool: get_dataset        # 二进制小工具
  version: CV_resnet50_v3        # 版本号含软件分类（类别 + 模型 + 版本）
```

实时出数（主机侧动态生成，互斥替换上面的 yaml）：

```yaml
data_source:
  realtime_tool: tools/gen_input.py
  seed: 42
  count: 100                     # 生成样本条数
```

> 模式通过字段存在性区分：写 `fetch_tool/version` 是静态，写 `realtime_tool` 是实时，不需要额外 `mode` 开关。

出数产物对**所有可执行平台共用**（同一份输入与 golden），这是跨平台精确比对的基础。

**平台归属：** fpga 走本章 4.1~4.7（6 步完整流程）；link / rtl2c / emu 流程见 [子 Wiki 1](./01_子wiki_版本制作与加载.md) 各对应章节 + 本章 4.8 简表；eda 仅打包不执行用例（[子 Wiki 1 · 第 4 章](./01_子wiki_版本制作与加载.md)）。

### 4.1 FPGA 平台主流程总览

FPGA 平台是 Autotest 最完整的执行路径，共 6 个步骤。其他平台的流程有各自特点，见 4.8 表格。

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 1. 文件上传   │ ──►│ 2. 触发版本加载│ ──►│ 3. 业务启动  │
└──────────────┘    └──────────────┘    └──────┬───────┘
                                               │
┌──────────────┐    ┌──────────────┐    ┌──────▼───────┐
│ 6. 用例切换  │ ◄──│ 5. 业务结果确认│ ◄──│ 4. 中间结果软调│
└──────┬───────┘    └──────────────┘    └──────────────┘
       │
       └─► 回到 3（软切换） 或 回到 2（硬切换）
```

**业务启动内部包含**：指导 CPU 桩 → 接口 FPGA、配置定时器、指导数据下发、配置数据比对逻辑。

**业务结果确认内部包含**：DFX 寄存器查询、精准比数（见子 Wiki 3）。

**用例切换前的可选扩展**：**阶段性比数**（自动化扩展功能），在切换前对本用例做分阶段精度/性能快照比对，见 [子 Wiki 2 · 第 5 节](./02_子wiki_中间结果软调.md)。这与用例切换可独立启用（阶段性比数默认关闭，性能用例或深度定位场景开启）。

**用例数据归档**：切换完成后自动归档本用例的日志、dump、DFX 快照、比对报告（见第 7 章）。

### 4.2 步骤 1：文件上传

**目的：** 将本次用例所需的所有产物上传到 CPU 桩或指定位置。

**上传内容：** 与 [子 Wiki 1 · 第 2 章](./01_子wiki_版本制作与加载.md) FPGA 版本组件对齐：

| 类别 | 说明 | 上传目标 |
|---|---|---|
| SDK | CMC 分发；可选上传或由业务软件裁剪内嵌 | CPU 桩指定目录（按需） |
| 平台软件 | 运行时核心 | CPU 桩指定目录 |
| 业务软件 | AI 业务逻辑（打包脚本来自 smartci 仓库） | CPU 桩指定目录 |
| 模型文件 | 编译器转换后产物 | CPU 桩指定目录 |
| CPU 桩固件 | 嵌入式芯片软件（按需刷写） | 嵌入式芯片 |
| 软调插件 | 内存读写插件 | CPU 桩指定目录 |
| 激励数据 | 出数产物（见 4.0.2） | CPU 桩数据目录 |
| Golden 数据 | 参考输出 | 主机侧（比对时使用） |
| 用例配置 | pytest 参数化数据 / yaml / json | CPU 桩或主机 |

**路径与命名规范：** 参考 **子 Wiki 1** 第 2 章。

**自动化实现：** Autotest 文件传输模块（基于 RDO 文件下发通道）。

**失败兜底：** 手动通过 scp/ftp 将文件传至 RDO 暂存区，再由 CPU 桩串口命令拉取。

### 4.3 步骤 2：触发版本加载

**目的：** 将已上传的 SDK / 平台软件 / 业务软件 / 模型加载到 CPU 桩与被测 FPGA。

**加载顺序：** SDK（可选）→ 平台软件 → 业务软件 → 模型（详见 **子 Wiki 1** 第 2 章 FPGA 加载）。

**校验项：**

- **版本指纹 VerFingerPrint 校验**：解析业务软件 DEBUG 日志中的版本指纹行，与期望一致
  - 日志格式：`VerFingerPrint[<组件>]: <指纹字符串>`
  - 示例：`VerFingerPrint[biz_sw]: 65010219930822401X`
  - 每个关键组件（SDK / 平台软件 / 业务软件 / CPU 桩 / 软调）各打印一条
- 加载日志出现成功关键字
- 关键寄存器状态正确

**失败兜底：** 串口手动敲加载命令，逐步定位失败阶段。

### 4.4 步骤 3：业务启动（核心）

#### 4.4.1 定义

业务启动 = 配置运行时参数 → 触发 **DO 加载**（DO = 业务软件 + 模型打包后的简称）→ 选择用例 → 触发业务激励。

#### 4.4.2 启动命令序列

```
[CPU 桩] 1. 初始化运行时环境、配置运行时参数（硬件规格 / 功耗模式等，见 4.4.4）
[CPU 桩] 2. 触发 DO 加载（业务软件 + 模型）
[CPU 桩] 3. 选择用例（运行模型 / 运行规格 / 比数模式 / 阶段比数模式）
[CPU 桩] 4. 配置 dump 开关（端到端 / 阶段性，见子 Wiki 2）
[CPU 桩] 5. 加载权重 / 激励 / GOLDEN
[CPU 桩] 6. 触发业务激励
[CPU 桩] 7. 查询状态直至完成
```

**原则：凡涉及 CPU 桩 ↔ 被测 FPGA 的交互（上表步骤 2~7），都存在两条实现路径**（详见 [子 Wiki 2 · 4.1](./02_子wiki_中间结果软调.md)）：

| 路径 | CPU 桩 ↔ 被测 FPGA 的交互方式 | 依赖 |
|---|---|---|
| **标准路径** | 经接口 FPGA 消息通道下发（配置 / 加载 / 激励 / 状态查询均走此通道；激励由接口 FPGA 按定时器定时发）| 消息通道可用 |
| **备选路径** | CPU 桩通过软调底软接口直接读写被测 FPGA 内存 / 全局变量（激励改自触发模式）| 软调的内存读写接口 |

消息通道通则走标准路径，否则退到备选路径。CPU 桩侧所有流程由 Autotest 编排。

#### 4.4.3 启动状态机

```
INIT ──► LOADING ──► READY ──► RUNNING ──► DONE
                        │           │
                        ▼           ▼
                      ERROR       ERROR
```

每个状态对应 CPU 桩特定寄存器值或日志关键字；Autotest 以轮询或事件方式获取。

#### 4.4.4 启动参数维度

启动参数包括原型版本、运行模型、运行规格、比数模式、阶段比数模式、硬件规格、功耗模式。每个维度的取值集与默认行为详见 [子 Wiki 2](./02_子wiki_中间结果软调.md) 与启动配置文件；执行路径（标准 / 备选）由 Autotest 运行时自动选，不是用例维度。

参数矩阵的实现走 pytest `@pytest.mark.parametrize` 展开（环境准备 / fixture / markers / 并行 / 报告 / 生命周期钩子全用 pytest 原生机制；超时和重试用 `@pytest.mark.timeout` 与 `@pytest.mark.flaky` 等装饰器，详见 Autotest 项目文档）。

#### 4.4.5 失败兜底

串口手动执行启动序列，逐条确认每条命令回显；重点关注步骤 2（DO 加载）、5（权重 / 激励 / GOLDEN 加载）、6（业务激励触发）的返回值。

### 4.5 步骤 4：业务中间结果软调

**目的：** 在推理过程中抓取中间张量，用于精度定位。

**AI 特性：** 这是 AI 验证区别于通用验证的核心能力入口，支持两种比数模式（见 [子 Wiki 2 · 第 1 章](./02_子wiki_中间结果软调.md)）：

- **端到端比数**：使用原始业务软件，只看最终输出；对性能零扰动；常规回归与性能用例默认
- **阶段性比数**：业务软件配合导出部分中间张量，由接口 FPGA 按预先规划的 ID 比对 GOLDEN；精度定位与首用例准入用

模式切换策略（精度 vs 性能）、业务软件三个版本（初级 / 进阶 / 商用）、首用例机制与 dump 机制的完整细节见 **[子 Wiki 2](./02_子wiki_中间结果软调.md)**。

**失败兜底：** 通过软调接口读取被测 FPGA 对应内存地址做 dump，在主机侧用 compare 模块离线比对（即手动走**备选比数路径**，见 [子 Wiki 2 · 4.1.2](./02_子wiki_中间结果软调.md)）。

### 4.6 步骤 5：业务结果确认

**目的：** 对最终输出、性能指标、硬件状态进行比对与判定，给出 Pass/Fail/Warning。

**主要工作：**

- 与 golden 数据精准比数（bit 级完全一致，见子 Wiki 2 · 第 7 章）
- 与性能基线比对
- DFX 告警寄存器检查（见 [子 Wiki 4](./04_子wiki_DFX告警寄存器查询.md)）

**详细规则、比对指标、辅助分析手段，见 [子 Wiki 3](./03_子wiki_业务结果确认.md)。**

**失败兜底：** 手动下载结果文件到主机，使用离线比对脚本。

### 4.7 步骤 6：用例切换（核心）

#### 4.7.1 切换的挑战

AI 用例间会残留：输入/输出 buffer、模型权重内存、状态寄存器、DMA 配置、中断状态等。若不清理，下一个用例结果不可信。

#### 4.7.2 切换策略

| 策略 | 动作 | 耗时 | 适用场景 |
|---|---|---|---|
| 软切换 | 业务软件不动，切换模型 / 比数模式 / 运行规格等 | 秒级 | 同一份业务软件 DO 下的用例切换 |
| 中切换 | 切换业务软件 DO（支持卸载与加载，卸载可选） | 十秒级 | 跨业务软件 DO 的用例切换；阶段比数模式切换（初级 / 进阶 / 商用，需换 DO）走此档 |
| 硬切换 | FPGA 复位，重新加载全部版本（SDK / 平台软件 / 业务软件 / FPGA 镜像） | 分钟级 | 跨基线、异常恢复 |

**决策规则（简化）：**

```
if 换基线 (SDK / 平台软件 / FPGA 镜像):   硬切换
elif 换业务软件:                          中切换
else (换模型 / 规格 / 比数模式):          软切换
```

#### 4.7.3 切换命令序列（以中切换为例）

```
1. 停止当前业务激励（优雅停止）
2. 保存用例快照（日志、寄存器、关键 dump）
3. 清理 buffer 与中间状态
4. 卸载当前业务软件（可选）
5. 加载下一业务软件
6. 清零状态寄存器
7. 进入下一用例步骤 3（业务启动）
```

#### 4.7.4 切换点快照

切换前保存：

- Autotest 日志（本用例全量）
- CPU 桩日志
- 被测 FPGA 关键寄存器 dump
- 最后一次中间结果（如有）
- DFX 告警寄存器快照（见 **子 Wiki 4**）

#### 4.7.5 用例间依赖与资源调度

待单独成文（资源调度子 Wiki，TBD）：用例依赖（独立 / 链式 / 分组）、多 FPGA 板与 emu 实例的资源调度服务。可参考业界 LAVA（Linaro Automated Validation Architecture）模式。

#### 4.7.6 批量调度优化

为降低总耗时，Autotest 调度器按以下规则排序用例：

1. 优先按**版本基线**分组（减少硬切换次数）
2. 组内按**模型分类**（CV / NLP / 多模态等）分组（同类业务软件用例聚一起，减少中切换次数）
3. 组内按**模型**分组
4. 组内再按**运行规格**分组
5. 执行时按**比数模式**动态软切换

#### 4.7.7 失败兜底

切换异常时：
1. 手动复位 FPGA
2. 手动卸载平台软件 / 业务软件
3. 按第 2 步开始重新加载

### 4.8 各平台流程对照

本节以表格承载 5 个平台各自的执行流程，**每个平台的流程结构是独立的，并非"同一套 6 步在不同平台的变体"**。各平台版本加载细节见 [子 Wiki 1](./01_子wiki_版本制作与加载.md)；平台业务职责与需求见 [子 Wiki 5](./05_子wiki_多执行平台架构扩展.md)。

#### 4.8.1 各平台执行流程（简表）

| 平台 | 流程步骤（概述） |
|---|---|
| **link** | ① 加载出数 → ② 编译版本 → ③ 用功能仿真器执行 → ④ 结果导出比对 |
| **rtl2c** | ① 使用特殊脚本 → ② 编译版本 → ③ 在指定代码仓+工具上执行 → ④ 执行结果确认 → ⑤ 触发波形采集（耗时，调试态）**/** 子用例回归（常规态） |
| **fpga** | ① 编译版本 → ② 版本加载（版本+模型）→ ③ 用例执行（指导 CPU 桩→接口 FPGA、配置定时器、指导数据下发、配置数据比对逻辑）→ ④ 用例结果确认（DFX 寄存器、精准比数）→ ⑤ 触发阶段性比数（自动化扩展）/ 用例切换 → ⑥ 用例数据归档 |
| **emu** | 流程与 fpga 类似，但链路不同：走 EMU 自有 API → 通过 espi 控制 CPU 桩、通过 TESGINE 灌数据，**无接口 FPGA** |
| **eda** | ① 使用特殊脚本 → ② 编译版本 → ③ 打包版本 → ④ 上传版本（供人手动分析，不在 Autotest 中继续执行） |

#### 4.8.2 关键硬件链路差异

| 平台 | 命令通道 | 数据通道 | 是否经过接口 FPGA |
|---|---|---|---|
| fpga | Autotest → RDO → CPU 桩 | CPU 桩文件传输 | ✅ |
| emu | Autotest → EMU API → espi → CPU 桩 | TESGINE 直接灌入 | ❌ |
| link | 本地进程 stdio/API | 本地文件 | ❌ |
| rtl2c | 本地脚本调用 | 脚本带参 + 文件 | ❌ |
| eda | （Autotest 只负责打包） | — | — |

#### 4.8.3 用例级别 × 平台适用矩阵

| 用例级别 | link | rtl2c | fpga | emu | eda |
|---|---|---|---|---|---|
| **L0 冒烟** | ✅ 主力（准入） | ✅ | ✅ | ⚠️ 按需 | ❌ |
| **L1 基础回归** | ✅ | ✅ 主回归 | ✅ 主回归 | ⚠️ 按需 | ❌ |
| **L2 全量回归** | ✅ | ✅ | ✅ 主力 | ⚠️ FPGA 失败时 | ❌ |
| **L3 性能** | ❌ | ❌ | ⚠️ FPGA 不准 | ✅ **性能出口** | ❌ |
| **L3 长稳** | ❌ | ❌ | ✅ | ⚠️ 按需 | ❌ |
| **失败定位** | — | — | — | ✅ | ✅ 打包给人工 |

> ✅ 主力 / ✅ 可用 / ⚠️ 按需 / ❌ 不适用

**关键点：**
- **FPGA 性能不准**，性能用例的权威出口是 **emu**
- **eda 不参与回归**，只在需要波形分析时由 Autotest 打包上传给人手动

> 平台架构差异（跨平台 bit-exact、性能基线归属、EMU 链路、同 RTL commit 原则）见 [子 Wiki 5](./05_子wiki_多执行平台架构扩展.md) 与 [子 Wiki 6](./06_子wiki_Autotest软件架构.md)。

### 4.9 整体回归编排

5 个平台不是孤立使用的，Autotest 按如下流水线组织回归执行：

#### 4.9.1 流水线图

```
          ┌─────────────────────┐
          │ 公共前置：用例选择 + 出数 │
          └──────────┬──────────┘
                     │
                     ▼
              ┌───────────────────────┐
              │   link   ‖   rtl2c    │  ← 并行：link 准入 + rtl2c 波形
              └──────────┬────────────┘
                         │
                     link 通过?
                         │
                    ┌────┴────┐
                   否         是
                    │          │
                    ▼          ▼
                 阻断       ┌────────────┐
                (rtl2c       │   fpga     │  ← 主回归
                 波形供诊断)   └──────┬─────┘
                                    │
                               fpga 通过?
                                    │
                            ┌───────┴───────┐
                           是              否
                            │               │
                            ▼               ▼
                     ┌───────────┐   ┌──────────────────┐
                     │ 性能用例   │   │ 自动打包 eda +   │
                     │ → emu     │   │ emu 回归 + 自动  │  ← 诊断
                     │（性能出口）│   │ git bisect 定位  │
                     └─────┬─────┘   └────────┬─────────┘
                           │                  │
                           ▼                  ▼
                         归档           人手动分析（eda 波形）
```

#### 4.9.2 编排规则

| 阶段 | 触发条件 | 失败时行为 |
|---|---|---|
| 公共前置 | 每次回归起点 | 阻断整轮 |
| link ‖ rtl2c 并行 | 公共前置完成 | link 失败**阻断** fpga，避免浪费板级资源；rtl2c 失败时波形仍用于 bisect 定位 |
| fpga 主回归 | link 通过 | 进入诊断分支；与 rtl2c 交叉校验（任一失败另一路大概率同失败）|
| emu 性能出口 | fpga 通过 | 性能不达标视为回归失败 |
| eda 自动打包 | fpga / rtl2c 失败时**自动触发** | 上传至指定位置，通知责任人 |
| emu 诊断回归 | fpga / rtl2c 失败时**自动触发**，与 eda 并行 | 辅助定位；配合自动 git bisect 缩小首坏 commit 范围 |
| 归档 | 全流程结束 | — |

> 编排背后的设计决策（link 前置 / link ‖ rtl2c 并行 / emu 双重角色 / 失败自动 bisect 等）见 [子 Wiki 6 · ADR](./06_子wiki_Autotest软件架构.md)。

## 第 5 章　用例组织与分层

### 5.1 用例矩阵

用例维度与 4.7.6 批量调度分组规则对齐，支持任意维度筛选执行：

**维度 A：模型分类**

- CV（分类、检测、分割）
- NLP（文本分类、生成、理解）
- 多模态（图文、语音等）

**维度 B：模型**

- 具体模型名（如 resnet50、bert_base、clip）；多模型组合用 `x` 连接

**维度 C：运行规格**

- 首用例规格 / 单规格优化版本 / 规格版本（见 4.4.4）

**维度 D：用例类型**

- 功能 / 性能 / 精度 / 稳定性

**维度 E：执行阶段标签**

- 加载阶段 / 推理阶段 / 切换阶段

用 pytest markers 承载这些维度（见 4.4.4）：`@pytest.mark.cv`、`@pytest.mark.resnet50`、`@pytest.mark.spec_mini`、`@pytest.mark.accuracy` 等，筛选时 `pytest -m "cv and accuracy"`。

### 5.2 用例命名规范

```
{prototype_version}_{model_name}_{case_type}_{id}
```

字段说明（对齐 4.4.4 启动矩阵）：

- `prototype_version`：原型版本
- `model_name`：模型名；多模型组合用 `x` 连接
- `case_type`：functional / perf / accuracy / stability
- `id`：序号

示例：

- `proto_v1_resnet50_perf_001`
- `proto_v1_bert_base_accuracy_012`
- `proto_v2_resnet50xbert_base_stability_003`

不再在命名中编码精度模式（精度一律 bit 级一致，不再是可变维度）。

### 5.3 用例优先级

| 级别 | 含义 | 触发时机 |
|---|---|---|
| L0 | 冒烟 | 每次版本加载后，分钟级 |
| L1 | 基础回归 | 每日构建 |
| L2 | 全量回归 | 版本提测、周度 |
| L3 | 长稳/边界 | 月度、发布前 |

### 5.4 用例数据管理

- 输入数据：版本化存储于制品库
- Golden 数据：版本化，与模型版本绑定
- 校准集：用于量化场景，单独版本化

## 第 6 章　AI 验证能力扩展概述

本章概述 Autotest 在 AI 场景下的能力扩展方向，具体实现细节见各子 Wiki（主要在 **子 Wiki 2**）。

### 6.1 业界参考与共性能力

业界 AI 硬件自动化测试框架普遍具备：

| 能力 | 业界代表 | 说明 |
|---|---|---|
| 阶段性 dump | NVIDIA TensorRT profiler、华为 CANN msaccucmp、MLIR 生态 | 按阶段/算子粒度抓取中间张量 |
| 精度比对 | cosine/L1/L∞/ULP 等多指标 | 多数框架内置；MLPerf 有统一准入阈值规范 |
| 性能 profiling | Nsight、CANN profiler | 端到端与算子级延迟 |
| Golden 管理 | 业界常用 PyTorch / ONNX Runtime 等 reference runtime | 本项目用 **GC 硬件计算功能仿真**（bit 级一致要求决定）；版本化 golden |
| DFX/告警查询 | 通过驱动 sysfs、寄存器接口、专用诊断工具 | 见 **子 Wiki 4** |
| 异常注入 | 部分框架支持 | 当前阶段暂不扩展 |

### 6.2 当前 Autotest 能力差距

| 能力项 | 现状 | 扩展路径 |
|---|---|---|
| 最终输出比对 | 已有 | 继续保留，指标丰富化 |
| 阶段性 dump | **缺失** | 新增阶段性 dump（业务软件配合，三版本；见子 Wiki 2 · 第 4 章）|
| 精准比数（bit 级 + 辅助分析）| **缺失** | 新增 bit 级比对 + FAIL 时辅助分析（分 block / LCS 偏移 / 位域 / QSNR，见子 Wiki 2 · 第 7 章）|
| 性能用例阶段性比数回落 | **缺失** | 端到端失败时自动回落阶段性比数定位（子 Wiki 2 · 第 3 章）|
| Golden 版本化管理 | 弱 | GC 硬件计算功能仿真出 GOLDEN；基线 manifest 记录 `gc_version`；归档演进路线（git lfs / 服务器 / dbox / onebox）见子 Wiki 3 · 2.2 |
| DFX 告警查询 | 查询接口已有，比对和调度待建 | 新增比对层 + 在 Autotest 增加查询调度点（子 Wiki 4） |

### 6.3 扩展后主流程的变化

- **步骤 3 业务启动**：增加 dump 配置参数
- **步骤 4 中间结果软调**：从可选变为核心能力，支持阶段性 / 端到端两种比数模式
- **步骤 5 业务结果确认**：功能走 bit 级精准比数（见子 Wiki 2 · 第 7 章）+ 性能走基线偏差判定 + DFX 联合判定

## 第 7 章　结果分析与归档

### 7.1 日志规范

| 日志类别 | 内容 | 命名规则 |
|---|---|---|
| 用例主日志 | Autotest 全流程日志 | `{case_id}_{timestamp}.log` |
| CPU 桩日志 | 串口回显完整记录 | `{case_id}_cpustub_{timestamp}.log` |
| 比对报告 | 精度/性能比对 | `{case_id}_compare_{timestamp}.html` |
| DFX 快照 | 告警寄存器值 | `{case_id}_dfx_{timestamp}.json` |

### 7.2 报告格式

- 用例级报告：单用例 Pass/Fail + 指标
- 套件级报告：汇总 + 失败列表 + 趋势
- 模型级报告：按模型聚合精度/性能

### 7.3 归档

- 归档位置：制品库 `reports/{date}/{build}/`
- 保留策略：L0/L1 保留 30 天，L2 保留 90 天，L3 长期保留
- 失败用例全部附件长期保留

### 7.4 问题单关联

失败用例自动生成问题单草稿，字段：用例 ID、失败阶段、关键日志、DFX 快照、相关版本。
