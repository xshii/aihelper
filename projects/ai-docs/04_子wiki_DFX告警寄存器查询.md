# 子 Wiki 4：DFX 硬件通用告警寄存器查询

> DFX = 芯片侧的硬件告警寄存器集合（ECC / 总线 / 温度 / 降频 / DMA / 看门狗等）。
>
> **查询接口复用芯片已有的 DFX 接口**（不重新开发）；寄存器清单与接口语义统一在**某个网页查询**。
>
> 对应主文档 4.6 节；参与用例裁决的联合判定见 [子 Wiki 3 · 5.2](./03_子wiki_业务结果确认.md)。

---

## 1. 查询动作

Autotest 在用例前后调用**现有 DFX 接口**读一次寄存器值（单寄存器 / 批量 dump 都走原生接口），不开发新通道。寄存器地址、宽度、语义 → **查对应网页**（规格随芯片版本更新，不在此文档维护）。

## 2. 关键动作

### 2.1 清零

- 错误计数型寄存器多数是 **read-clear**（读即归零）
- 少数是**写清**（显式写 1 到对应位）
- 每个用例前 / 版本加载后**主动清零一次**，避免历史累积污染本次判定
- 读后的值含义是「**本用例期间新增的告警**」，不是总累积

### 2.2 异常判断

对每个寄存器按语义做判断（语义见网页），两大类：

| 类型 | 规则 |
|---|---|
| 计数型（ECC / DMA 错误等）| `value > 0` 即告警 |
| 状态型（总线 / 看门狗 / 降频等）| 按特定位 / 阈值判告警，参考网页 check 规则 |

判定按严重级分档（映射在寄存器元数据里，与网页同步）：

| 等级 | 影响 |
|---|---|
| 阻断 | 用例 **FAIL**（即使精度 / 性能都过）|
| 非阻断 | 用例 **WARN** |
| 仅记录 | 不影响裁决（如正常范围内的降频计数）|

## 3. 查询时机

| 时机 | 动作 |
|---|---|
| 版本加载后 | 清零 + 读一次（确认干净状态；不干净则后续用例不可信）|
| 用例结束前 | 读一次（参与 [03 · 5.2](./03_子wiki_业务结果确认.md) 联合判定）|

性能用例**不做用例中周期轮询**（干扰计时），仅两个端点读。

## 4. 失败兜底

| 场景 | 处置 |
|---|---|
| Autotest 查询异常 | 手动串口调现有 DFX 接口读关键寄存器，结果手动贴到报告 |
| 所有寄存器读出一致异常值（如 `0xDEADBEEF`）| CPU 桩 / 总线未就绪，回到 [01](./01_子wiki_版本制作与加载.md) 排查加载 |
| 语义对不上 | 查寄存器元数据网页（注意芯片版本要与当前基线对齐）|

## 5. 常见问题

| 现象 | 解读 |
|---|---|
| 加载后基线就有告警 | 上一用例未清零或硬件真异常；先强制清零再看 |
| 精度性能全过，但 ECC SBE=1 | 单比特被自动纠正，结果可信，判 WARN；持续增长需换板 |
| 性能用例大面积 WARN（降频）| 散热 / 环境温度问题，当轮性能不作基线 |
| 看门狗超时但精度 PASS | 返回的是旧值 / 默认值，不可信，判 FAIL |
