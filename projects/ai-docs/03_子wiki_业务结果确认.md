## 2. 功能结果确认

### 2.1 比对原则：bit 级一致

功能结果采用**精准比数**，唯一通过标准是 **bit 级一致**。FAIL 时附带辅助分析（分 block / LCS 偏移 / 位域 / QSNR）提升定位效率，**辅助结果不改变裁决**。

详细比对机制、辅助分析手段、两条比数路径（标准 / 备选）见 **[子 Wiki 2 · 第 7 章](./02_子wiki_中间结果软调.md)**。本章聚焦 GOLDEN 数据与出数流程。

### 2.2 GOLDEN 数据

#### 2.2.1 来源：GC 硬件计算功能仿真

GOLDEN **来自 GC 硬件计算功能仿真**（GC = 硬件计算的功能级参考实现，仿真硅片的算子级行为），**不使用 CPU / GPU 参考**。CPU / GPU 参考的 reduction 顺序、中间精度、舍入模式、量化尺度都和硬件不同，无法产出 bit 级一致输出；GC 仿真覆盖硬件所有计算细节，GOLDEN 即"硬件该产出什么"的权威参照。

#### 2.2.2 GC 版本必须与被测 FPGA 版本匹配

**每个基线都必须明确"当前被测 FPGA 对应的 GC 版本"**，不能自动取 GC 最新版。

| 情况 | 选择 |
|---|---|
| FPGA 发布版本 A；GC 也在 A | 直接用 A |
| FPGA 发布版本 A；GC 已 bugfix 到 next 版本 B | **继续用 A** —— 当前 FPGA 还没吃到 B 的 fix，用 B 出的 GOLDEN 会对不齐 |
| FPGA 升级到含 B 的新版本 | 切换到 B 的 GOLDEN |

**基线 manifest** 里显式记录 GC 版本字段，与其他平台版本一起做 `baseline_id` 一致性校验（参考 [子 Wiki 1 · 第 7 章](./01_子wiki_版本制作与加载.md)）：

```yaml
baseline_id: v1.2.3-20260421-a1b2c3d
rtl_commit: abc1234
gc_version: gc_v1.2.3          # ← 当前被测 FPGA 对应的 GC 版本
platform_packages:
  ...
```

#### 2.2.3 存储格式：硬件目标格式（非 double）

GOLDEN **以硬件目标格式存储**（与被测 FPGA 实际产出的 bit 流完全一致），**不是 double / FP32 等通用浮点格式**。

为什么：

- bit 级比数要求 actual 与 golden 的底层 bit 流完全一致
- 转 double 存储会引入格式转换，破坏 bit-exact 比较
- 直接存硬件目标格式，接口 FPGA / compare 模块读入即用，零转换

> 注：辅助分析中的 QSNR（见子 Wiki 2 · 7.7）内部会把双方临时转 double 计算信噪比——那是**定位手段**而非存储格式，不影响 bit 级裁决。

#### 2.2.4 归档策略（演进中）

GOLDEN 原始数据量较大，归档方案演进中：

| 方案 | 状态 | 说明 |
|---|---|---|
| **git lfs 上传** | 候选 | 与代码仓同版本化，便于追溯；需评估 lfs 容量与带宽 |
| **服务器托管** | 候选 | 自建服务器按 `baseline_id` 托管，作为日常快速访问缓存 |
| **dbox**（网盘）| 最终归档位置 | 当前**无自动化上传下载接口**，人工放 |
| **onebox**（另一网盘）| 待调研 | **需研究是否有自动化上传下载接口**；若有，优先作为自动化归档入口 |

**当前为过渡状态**（dbox 人工 + 服务器 / git lfs 缓存），不是最终方案。等 onebox 自动化接口调研完再定型——届时所有流程都应走自动化通道，人工归档只作冷备份。

### 2.3 出数

#### 2.3.1 当前：静态出数

- 出好数后**固化存储**，用例执行时直接引用
- 用版本号 + 软件分类标识（详见主文档 4.0.2 `data_source` 配置）
- 优点：确定性高，多次执行输入一致，回归可复现
- 缺点：新增用例或参数化输入需先跑出数 pipeline 再归档

#### 2.3.2 演进：实时出数

- 后续 Autotest 支持运行时动态调用出数工具生成
- 适用：随机 / 参数化输入、CI 触发时按配置临时生成

#### 2.3.3 出数仓与比数仓同仓管理

**出数工具与 compare 比数模块（见 [子 Wiki 2 · 8.4](./02_子wiki_中间结果软调.md)）放在同一独立仓库管理**。理由：

- 共用底层数据格式、硬件目标类型定义、GOLDEN 索引规范
- 避免跨仓维护导致格式不一致
- 跨平台复用（link / rtl2c / fpga / emu）入口统一

## 3. 性能结果确认

### 3.1 采集方式：解析 DEBUG 日志

性能数据**由平台软件 / 业务软件打印到 DEBUG 日志**，Autotest 驱动 CPU 桩读取日志文件后 dump 到主机做离线解析。不依赖额外的性能探针，数据就在日志里。

**标准日志格式：**

```
[PERF]ID=<id>, ROUND=<round>, x_cycle=<cycle_count>, y_cycle=<cycle_count>, ...
```

- `[PERF]` 是日志前缀标识，解析器按此抓取
- `ID`：性能测量点 ID（每个关键阶段 / 算子簇可分配一个）
- `ROUND`：多轮执行时的轮次编号
- `x_cycle` / `y_cycle` / ...：各阶段的硬件 cycle 计数（可扩展多列，对应不同测量点）

**采集流程：**

```
平台软件 / 业务软件执行期 → 打 [PERF] 日志到 CPU 桩
       ↓
Autotest 驱动 CPU 桩把日志文件 dump 回主机
       ↓
主机侧解析器按 [PERF] 前缀抓取，按 ID + ROUND 聚合
       ↓
产出性能指标（见 3.2）+ 对比基线（见 3.3）
```

**好处：**

- 零扰动（DEBUG 日志在平台软件本就存在，不是为 Autotest 专门加的埋点）
- 可追溯（日志与裁决一起归档）
- 格式稳定（`[PERF]` 前缀 + kv 对，解析器不需要平台特例）

### 3.2 性能指标

| 指标 | 定义 | 日志字段（示例） |
|---|---|---|
| End-to-end latency | 输入就绪到输出就绪的时间 | `x_cycle`（端到端计数）|
| Pure inference latency | 纯硬件推理时间 | `y_cycle`（硬件核心计数）|
| Throughput | 单位时间完成推理数 | 多 ROUND 聚合计算 |
| P99 / P999 latency | 长尾延迟 | 多 ROUND 分位数 |
| 功耗 | 推理稳态功耗 | 外部功耗探头（不在 [PERF] 日志内，独立通道）|

### 3.3 性能基线

每个模型维护性能基线：

```yaml
baseline:
  model: cv_resnet50
  hardware: fpga_vX
  batch_size: 1
  latency_ms:
    mean: 12.5
    p99: 14.0
  throughput_qps: 78
  updated_at: 2026-04-01
```

基线更新策略：

- 性能提升 >5%：人工评审后更新
- 性能下降：必须定位原因，不得直接更新基线
- 硬件变更或重要优化后：必须重建基线

### 3.4 判定阈值

```
for metric in [latency, throughput, p99]:
    actual = measure()
    baseline = load_baseline()
    deviation = (actual - baseline) / baseline
    if deviation 超阈值:
        mark_fail(metric)
```

默认阈值：

- latency / p99：+5% 内通过，+5%~+10% WARN，>+10% FAIL
- throughput：-5% 内通过，-5%~-10% WARN，< -10% FAIL

> 注：性能指标是模拟量测量，抖动不可避免，所以有偏差阈值。这与功能结果的 **bit 级一致**不冲突——精度走 bit，性能走模拟量偏差。

### 3.5 性能波动处理

- 每个用例执行 N 次（建议 N=5），取中位数与 P99
- 首次 warmup 运行不计入
- 连续失败 3 次才正式判 FAIL，1~2 次失败标记 FLAKY

### 3.6 平台选择

**性能用例的权威出口是 emu，不是 fpga**（FPGA 测性能不准，见主文档 4.0.1 / 4.9）：

- fpga 仅用于"功能结果确认"阶段同时抓取的性能参考数据
- 最终性能判定以 emu 为准

## 5. 裁决规则与报告

### 5.1 三态裁决

| 状态 | 含义 |
|---|---|
| PASS | 功能、性能、DFX 全部通过 |
| WARN | 主体通过但存在非阻断告警或 FLAKY |
| FAIL | 任一项不达标或有阻断告警 |

> **功能裁决不存在 WARN 状态**：bit 一致即 PASS，bit 不一致即 FAIL，辅助分析不改判。WARN 只来自性能或 DFX。

### 5.2 联合判定表

| 功能 | 性能 | DFX | 结果 |
|---|---|---|---|
| PASS | PASS | 无告警 | **PASS** |
| PASS | PASS | 非阻断告警 | **WARN** |
| PASS | PASS | 阻断告警 | **FAIL** |
| PASS | WARN | 无告警 | **WARN** |
| FAIL | * | * | **FAIL** |
| * | FAIL | * | **FAIL** |
| * | * | 阻断告警 | **FAIL** |

### 5.3 报告结构

**用例级报告**（HTML + JSON 双份）：

```
报告标题：<case_id>
├─ 概要：结果、耗时、时间戳
├─ 功能：bit 比对摘要 + 辅助分析报告（FAIL 时，见子 Wiki 2 · 7.9）
├─ 性能：各指标值、与基线对比、趋势
├─ DFX：告警列表、寄存器快照
├─ 中间结果：首发散阶段（如有，见子 Wiki 2 · 7.12）
├─ 日志：Autotest 日志、CPU 桩日志、链接
└─ 归档位置（含 GOLDEN 版本、GC 版本引用）
```

**套件级报告**：

```
套件：<suite_id>
├─ 汇总：PASS/WARN/FAIL 数量与比例
├─ 失败列表：case_id + 一行原因
├─ FLAKY 列表
├─ 性能趋势：对比前 N 次构建
└─ 下钻链接 → 各用例报告
```

### 5.4 上报与归档

- 用例完成即时上报裁决结果到 Autotest 平台
- 套件结束生成汇总报告
- 归档按主文档 7.3 节
- FAIL 用例自动生成问题单（含功能 / 性能 / DFX 摘要，以及 GC 版本 + baseline 版本）

## 6. 失败兜底

| 失败场景 | 手动步骤 |
|---|---|
| 最终输出未回传 | 串口手动下发 dump 最终输出命令，从 CPU 桩取（即手动走**备选比数路径**，见 [子 Wiki 2 · 4.1.2](./02_子wiki_中间结果软调.md)）|
| compare 比数模块异常 | 本地用 Python 加载 dump / golden，调 compare 仓库的 `bit_compare` 手动跑 |
| Golden 缺失 | 按 2.2.2 确认当前 FPGA 对应的 GC 版本，从服务器 / git lfs / dbox 拉取；若还缺，GC 仿真临时补跑后入库 |
| GC 版本错配（大面积 bit 不一致）| 检查基线 manifest 的 `gc_version` 与 FPGA 版本的匹配关系（见 2.2.2），回切到正确版本 |
| 性能日志未打出或格式错 | 确认平台软件 / 业务软件的 [PERF] 打印开关；按 3.1 格式校对 `ID / ROUND / *_cycle` 字段；必要时手动从 CPU 桩 dump 日志文件离线解析 |
| 性能数值异常（抖动 / 飙高）| 关闭所有 dump、延长 warmup、增加采样轮次（ROUND）重测；必要时切到 emu 平台复测 |
| DFX 查询不到 | 见 [子 Wiki 4](./04_子wiki_DFX告警寄存器查询.md) 手动查询步骤 |

## 7. 常见问题

| 现象 | 原因 | 处置 |
|---|---|---|
| 功能比对永远通过但输出全零 | 输入未注入成功（bit 级相等但语义无效）| compare 模块加"非零占比"健壮性检查 + 上游输入自检 |
| 功能 FAIL 但辅助报告显示 QSNR 极高 | 少量低位差异，bit 仍不一致仍判 FAIL | 按子 Wiki 2 · 7.10 综合定位建议排查累加顺序 / 舍入 |
| 功能 FAIL 且 LCS 匹配率接近总长但 offset ≠ 0 | 整体地址 / 布局错位 | 查 DMA 起址、padding、layout 转换 |
| GC 版本错配导致大面积 FAIL | 错用了 GC next 版本的 GOLDEN 对当前 FPGA | 回切到 FPGA 发布版本对应的 GC GOLDEN（2.2.2）|
| GOLDEN 存成 double 导致 bit 比对永远 FAIL | 格式错用 —— GOLDEN 必须硬件目标格式 | 2.2.3 明确要求；重新用 GC 仿真按硬件目标格式出 GOLDEN |
| GOLDEN 拉取慢、CI 超时 | 从 dbox 下载（无自动化接口）| 优先从服务器 / git lfs 拉缓存；dbox 仅作最终归档；等 onebox 自动化接口落地 |
| 性能偶发飙高 | 系统干扰 / DMA 冲突 / DFX 告警 | 联合 DFX 查询；提高采样数 |
| 功能 PASS 但性能 WARN | 性能优化特性开启后有抖动 | 查首发散点；确认开启的硬件优化配置 |
| 连续失败后突然通过 | 硬件热重置 / 偶发故障 | 必须做长稳回归，不能当"已修复" |
