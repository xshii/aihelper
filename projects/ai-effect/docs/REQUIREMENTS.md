# 算子调试与对照工具 — 需求文档

## 1. 背景

### 1.1 系统现状

公司内部有一款 AI 加速器硬件,模型通过 C 语言的 intrinsic 接口编写,通过自研编译器编译后运行在硬件上。

关键技术特征:

- **自研 C 编译器**(短期内不能修改/扩展)
- 代码中的硬件指令通过 **特殊宏定义** 表达(目前已知 5-6 种 DEFINE 宏)
- 宏展开后向硬件寄存器写值
- 每个算子有自己的接口(Softmax / LayerNorm 等不统一)
- 上层接口最终都收敛到那 5-6 种硬件宏的组合(链式调用)

### 1.2 当前痛点

1. **比对盲区**:模型推理流程很长,最终结果错了之后,中间各环节的输出无法对照,无法定位
2. **流程难追踪**:执行流程复杂,出问题时不知道是哪个算子哪一步
3. **缺乏日志**:没有结构化的日志/trace 体系,无法回溯
4. **无离线对照**:出问题时没有 reference 实现作为 golden

## 2. 目标

构建一套 **算子调试与对照工具**,要求:

- 模型/算子开发者**几乎无感知**(零代码改动,开关式启用)
- 出问题能**自动定位**到第一个发散的算子
- 工具栈用**业界主流技术**(Clang AST + Python),内部 LLM 能理解、能协助开发
- 架构**支持长期演进**,从插桩工具扩展到内存分析、依赖可视化等多种工具
- **不依赖修改自研编译器**

## 3. 功能需求

### FR-1 自动插桩(免适配)

工具自动识别源码中的算子接口调用和硬件宏调用,在调用前后插入 hook 调用。

- FR-1.1 算子开发者**不需要修改源码**
- FR-1.2 通过编译选项或环境变量启用/关闭
- FR-1.3 不启用时**零开销**(原始源码不变)
- FR-1.4 识别 5-6 种硬件 DEFINE 宏(数量可扩展)
- FR-1.5 自动解析算子接口的入参结构体(基于 Clang 类型信息)
- FR-1.6 支持声明式规则描述新增的宏/接口(规则用 Python 装饰器写)
- FR-1.7 支持黑名单(某些函数/文件不插桩)

### FR-2 Hook 运行时库

hook 函数被调用时的行为可配置。

- FR-2.1 支持以下模式(每个 hook 点可独立配置):
  - `DUMP_AND_RUN`:dump 中间数据 + 执行原始指令(默认)
  - `DUMP_AND_SKIP`:dump 中间数据 + 跳过原始指令(用 reference 输出替换)
  - `REPLACE_WITH_REF`:不执行原始指令,只跑 reference 实现
  - `HASH_ONLY`:只记录数据 hash,不导出全量
  - `PASSTHROUGH`:不做任何事(关闭某点)
- FR-2.2 数据导出支持 **DMA 机制**,把片上数据搬移到 host 内存
- FR-2.3 落盘格式结构化(JSONL + 关联二进制),带元数据
- FR-2.4 trace 元数据包含:op_id、layer、iteration、调用栈、shape、dtype、参数值

### FR-3 Reference 实现库

host 侧的算法等价实现,作为 golden。

- FR-3.1 覆盖那 5-6 种硬件宏(每种至少一个 reference)
- FR-3.2 用主流库实现(NumPy / Eigen),易读易维护
- FR-3.3 跑得快(用于线上 replay 场景,不能慢几百倍)
- FR-3.4 确定性(同样输入,同样输出)
- FR-3.5 版本和硬件实现绑定(防止漂移)
- FR-3.6 接口和 hook 库一致,可作为 `REPLACE_WITH_REF` 模式的后端

### FR-4 离线对照分析

对比硬件 trace 和 reference,定位首个发散点。

- FR-4.1 按 op_id 对齐
- FR-4.2 多种容差比对(abs / rel / cosine / max diff)
- FR-4.3 自动报告 **第一个发散超过阈值的算子**
- FR-4.4 区分"上游传染"和"本算子出错"(看输入是否一致 + 输出是否一致)
- FR-4.5 产出结构化报告(JSON)+ HTML 可视化

### FR-5 编排入口

统一 CLI 工具,串起完整流程。

- FR-5.1 一条命令完成:插桩 → 编译 → 运行 → 收集 → 对照 → 报告
- FR-5.2 配置文件描述运行选项(算子选择、hook 模式、阈值等)
- FR-5.3 单步可分别调用

### FR-6 (后续)内存分析与依赖可视化

基于 Clang AST 分析内存编排和算子依赖(V2+,本期可选)。

- FR-6.1 识别 buffer 分配、复用、生命周期
- FR-6.2 检测内存重叠 / 编排合理性
- FR-6.3 构建算子间依赖图 + 时序提交图
- FR-6.4 可视化(Graphviz / Mermaid / D3)

## 4. 非功能需求

- **NFR-1 免适配 / 用户无感知**:零代码改动;编译选项/环境变量启用;不启用时无副作用、无性能影响
- **NFR-2 长期演进**:架构支持新增应用层模块,不影响底层;规则可扩展(声明式);版本化的 IR / 规则 / 报告格式
- **NFR-3 AI 友好**:主流技术栈(Clang AST + Python);不引入小众工具;规则用 Python 装饰器;清晰 schema 文档;单元测试齐全
- **NFR-4 软件架构合理**:单向依赖无循环;高内聚低耦合;每层单一职责;跨层只通过 IR / 接口通信
- **NFR-5 可测**:每层独立可测;Pattern 规则配 fixture(输入 .c / 期望 .c);集成测试不依赖真硬件

## 5. 约束

- **C-1 技术栈**:自研编译器不可修改;必须用 Clang AST(libclang)+ Python;不用小众工具(Coccinelle 机制可参考不可直接用)
- **C-2 源码**:源码含自定义 intrinsic / 宏,Clang 默认不认识;需要 **stub header** 让 Clang 能解析
- **C-3 运行环境**:仿真平台和真实硬件平台都要支持;线上场景也要能对照 reference(DMA 把数据搬到 host)

## 6. 用户场景

- **UC-1 模型推理结果错误**:跑 `diff`,工具编译插桩版 → 跑硬件 → 离线 reference 对照 → 报告"第一个发散的算子是 `conv_layer3 (iter 5)`,输入一致但输出 abs_diff=0.3"
- **UC-2 算子级 bisection**:用 `REPLACE_WITH_REF` 跳过若干算子,看结果是否正确,缩小问题范围
- **UC-3 线上问题对照**:采集线上输入(DMA dump),带回离线用 reference 跑,验证应该是什么结果
- **UC-4 (后续)内存编排检查**:`memcheck` 子命令静态分析报告可疑重叠

## 7. 成功标准

### V0 MVP(3 个月内)
- 能识别一种简单算子的硬件宏调用
- 完整跑通:插桩 → 运行 → reference 对照 → 报告
- 一个典型算子的 reference 实现
- CLI 可调用
- PoC 验证 Clang AST + 宏处理可行性

### V1(6 个月)
- 覆盖 5-6 种硬件宏全部
- 真实模型端到端跑通
- HTML 可视化报告

### V2(后续)
- 内存分析 / 依赖可视化
- Plugin 机制
- 线上对照场景验证

## 8. 范围外

- 不做 GPU/CPU 等其他硬件后端
- 不替代自研编译器
- 不做性能优化(只做正确性对照)
- 不在生产模型代码里持续保留 hook(只在 debug 模式下注入)
